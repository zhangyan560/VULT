# 印尼现金贷风控体系

## 市场特点

印尼是东南亚最大的现金贷市场之一。核心特点：

| 特点 | 说明 |
|------|------|
| **代办中介问题严重** | 中介远程控制用户手机完成贷款申请并抽佣，是埋点反欺诈特征设计的核心驱动力 |
| **多头借贷泛滥** | 用户在多个平台同时借款、以贷养贷，AFPI/CBI查询次数是重要风险信号 |
| **征信基础设施薄弱** | 央行征信（CBI）覆盖不全，需多个数据源互补（AFPI、IZI、ADVANCE、同盾电信多头等） |
| **产品结构多样** | 多种期数（1/1.5/2/3/4/6/9/12期）x 多种利率组合的产品矩阵 |
| **催收难度大** | 失联率高，催收效率是重要痛点，需要前置预测和策略调整 |

> [!info] 关键市场趋势
> - 白户占比从 86%（2024.8）降至 33%（2025.6），非白户逐渐成为主力
> - 12期产品占比从 0 快速增长到 6.3%（2025.10），额度使用率约 55%
> - 25年10月后下单率明显下降（76% → 66%）
> - 9期以上产品需做千2及以下利率（合规要求）

## 风控架构（一次/二次）

采用**两次风控**架构，核心逻辑是分阶段、分层过滤：

| 风控阶段 | 触发时机 | 基准时间点 | 核心作用 | 可用数据 |
|---------|---------|-----------|---------|---------|
| **一次风控** | 用户首次提交申请 | `first_done_ts` | 准入决策（能不能借） | 外部征信、设备信息、App列表、短时埋点 |
| **二次风控** | 通过一次风控后、实际放款前 | `ordertrace_created_ts` | 额度/定价决策（借多少、什么条件） | 一次风控数据 + 更多App内交互行为 |

```
用户申请 → 一次风控（准入） → 二次风控（额度/定价） → 放款 → 贷后催收
              |                    |                          |
          first_done_ts       ordertrace_created_ts       失联预测
          该不该借？            借多少、什么条件？           催收优先级
```

## 客群分层

### 5个维度的客群分层

风控策略基于以下5个维度对客群进行精细化分层，每个维度对应不同的风控差异：

| 维度 | 分层 | 风控差异 |
|------|------|---------|
| **借贷阶段** | 首贷 vs 复贷 | 首贷无历史行为，更依赖外部征信；复贷有内部还款表现可用 |
| **业务渠道** | 主营 vs 回捞 | 主营是正常审批流；回捞是被拒用户的二次筛选，风险更高 |
| **信用状态** | 白户 vs 非白户 | 白户占比从86%（2024.8）降至33%（2025.6），非白户逐渐成为主力 |
| **多头等级** | 1-4级 | 多头等级越高风险越大，不同评级中多头分布差异大 |
| **评级** | A-T（12档） | A-C逾期率5-10%，F-G逾期率10-17% |

### 借贷阶段：首贷 vs 复贷

| 对比项 | 首贷 | 复贷 |
|--------|------|------|
| 可用数据 | 外部征信（CBI/AFPI/IZI）、设备信息、App列表 | 以上全部 + 内部历史还款表现 |
| 建模策略 | 更依赖外部数据，特征稀疏问题更突出 | 可用历史子分（1/3/6/9个月表现）大幅提升模型效果 |
| 白户问题 | 白户比例更高，征信覆盖不足 | 有还款记录，白/非白问题相对弱化 |

### 业务渠道：主营 vs 回捞

| 对比项 | 主营 | 回捞 |
|--------|------|------|
| 定义 | 正常申请渠道 | 被一次风控拒绝后通过特定策略再次触达 |
| 风险特征 | 标准风险分布 | 整体风险更高（已被主营拒绝过） |
| 模型策略 | 主评级模型标准应用 | 需要专项二次捞回模型（如二次FPD31模型） |

### 信用状态：白户 vs 非白户

> [!info] 白户占比趋势
> 白户占比从 **86%（2024.8）** 降至 **33%（2025.6）**，非白户逐渐成为主力客群。这一变化对特征工程和建模策略有重大影响。

| 对比项 | 白户 | 非白户 |
|--------|------|--------|
| 定义 | 在CBI/AFPI无贷款记录 | 有征信记录 |
| 逾期率 | 显著低于非白户，但MOB增速更快 | 较高，但有更多征信特征可用 |
| 模型增益 | 二次风控FPD31模型中白户增益更大（白户vs非白户增益差约1pp） | — |

### 多头等级：1-4级

| 多头等级 | 含义 | 风险 |
|---------|------|------|
| 1级 | 多头借贷程度最低 | 最低 |
| 2级 | 轻度多头 | 较低 |
| 3级 | 中度多头 | 较高 |
| 4级 | 重度多头，高频跨平台借贷 | 最高 |

> [!info] 多头分布特征
> 不同评级中多头分布差异大。通过AFPI/IZI查询次数、CBI机构数量等特征识别多头等级。

### 评级体系：A-T（12档）

评级是主评级模型（V13 MOB1/MOB3）输出分箱后的业务划分，从A（最优）到T（最差）共12档：

| 评级区间 | 逾期率范围 | 风险描述 |
|---------|-----------|---------|
| A-C | 5%-10% | 优质客群，逾期率稳定 |
| D | — | D评级为特殊区间，纯特征失联模型可识别出风险≈R评级的人群（占完件千2） |
| E-G | 10%-17% | 中等风险 |
| H-T | 17%+ | 高风险客群 |

> [!info] 评级数据来源
> 来源：V13前期分析.xlsx — `by评级` Sheet，覆盖 A~T 评级的人数/金额/逾期率时间趋势（2023H1~2025.10）

#### 评级的建模用途

| 应用场景 | 说明 |
|---------|------|
| 历史子分建模 | 按评级分组切分（如头部/D评级 x 短期/长期），捕捉评级内部差异 |
| 失联模型 | E-G评级中用纯特征失联模型找高风险子群（≈R评级风险，占完件千2） |
| 交叉分析 | `by评级+期数` 交叉分析各评级在不同期数的人数/金额/额度使用率 |
| 额度策略 | 高评级放更高额度，低评级限额 |

## 标签体系

### 各标签定义

模型训练的Y值（目标变量）由以下标签体系构成：

| 标签 | 定义 | 建模用途 |
|------|------|---------|
| **FPD7** | 首期还款日逾期7天 | 短期违约预警 |
| **FPD31** | 首期还款日逾期31天 | 主建模标签之一；二次风控FPD31模型专用Y |
| **1PD31** | 第一期逾期31天 | 等效于FPD31 |
| **MOB1 7+** | 放款后第1个月逾期7天+ | V13短期违约主Y；一次风控主要目标 |
| **MOB3 7+** | 放款后第3个月逾期7天+ | V13中期违约主Y；3期及以上样本使用 |
| **失联标签** | MOB1逾期后7天内联系>=5天未联系上（电话+WhatsApp） | 失联模型专用Y |
| **下单标签** | 7日内是否提现下单 | 双塔模型下单塔的Y值 |

> [!info] V13 双Y值设计
> V13主评级模型采用**双Y值并行建模**策略：
> - **MOB1 7+（短期）**：用全量样本训练
> - **MOB3 7+（中期）**：仅用3期及以上样本训练
>
> 最终融合两个模型的输出，兼顾短期和中期违约预测能力。

### 成熟期计算

标签需要"成熟"后才可用于建模，这是PIT设计的核心时间基础：

```
mob1_7_mature_date = first_done_date + 30天
```

- **物理含义**：MOB1 7+标签需要等到放款后第30天才能确定是否逾期7天
- **建模意义**：只有 `mob1_7_mature_date <= 样本截止日` 的样本，其MOB1标签才是"已成熟"的，可以纳入训练集
- **PIT防泄漏**：在App图网络建图时，严格用 `mob1_7_mature_date <= target_date` 过滤，防止未来逾期信息穿越到特征中

> [!tip] 成熟期的工程含义
> 成熟期设计是整个风控体系中防止数据穿越（Data Leakage）的关键机制。所有以逾期标签为基础的特征（App图网络的坏账率、历史子分的分箱等）都必须基于已成熟的标签计算，否则会导致模型在回测中虚高、上线后衰减。

| 标签 | 成熟期计算 | 说明 |
|------|-----------|------|
| FPD7 | `first_done_date + 首期还款日 + 7天` | 首期还款日因期数不同而不同 |
| FPD31 | `first_done_date + 首期还款日 + 31天` | 主建模标签，成熟期约30-60天 |
| MOB1 7+ | `first_done_date + 30天 + 7天` | 标准：放款后37天成熟 |
| MOB3 7+ | `first_done_date + 90天 + 7天` | 成熟期约97天，需3期及以上产品 |
| 失联标签 | MOB1逾期后再等7天 | 成熟期在MOB1基础上再加7天 |

### 时间窗口与OOT设计

| 窗口类型 | 时间范围 | 用途 |
|---------|---------|------|
| 上层融合训练窗口 | 24.8.1 ~ 25.7.31 | 一次风控融合模型INS |
| 子分系列训练窗口 | 23.7.1 ~ 25.6.30 | 各子分模型INS |
| OOT窗口 | 2025.8.01至今 | 模型线下评估 |
| 融合层样本拆分 | 80% INS + 20% OOS | 防止过拟合 |

> [!info] 为什么不同模型用不同训练窗口？
> - 融合层（主评级）用较新的24.8起始，保证学到最新的用户行为分布
> - 子分模型用更长的历史（23.7起），保证各分组（如按评级、按期数）有足够样本量
> - MOB3模型只用3期及以上样本，因为1/1.5/2期产品无法成熟到MOB3

### PIT设计的时间基础

PIT（Point-In-Time）是风控特征工程的核心原则，要求**特征计算时刻 = 决策时刻**：

| 阶段 | 时间锚点 | PIT要求 |
|------|---------|---------|
| 一次风控特征 | `first_done_ts` | 所有特征只能用 `first_done_ts` 之前的数据 |
| 二次风控特征 | `ordertrace_created_ts` | 可以额外使用申请到下单之间的埋点行为 |
| App图网络建图 | 每日独立快照 | 图中每条边/节点的标签只用已成熟的历史数据，不用当批次 |

## 外部数据源

### 6大数据源对比

印尼现金贷风控依赖6类外部数据源，互相补充覆盖征信薄弱地区的信息空白：

| 数据源 | 来源/类型 | 核心信息 | V13特征量级 |
|--------|---------|---------|------------|
| **CBI** | 央行征信（Bank Indonesia） | 信贷记录、逾期历史、机构查询次数、CBI评分、各产品类型（BNPL/CashLoan）债务状态 | 5,240个 |
| **AFPI** | 金融科技协会（P2P平台联盟） | P2P借贷记录、多头借贷、借款金额/期限/状态、机构类型（头部/AFDC/尾部） | 5,873个 |
| **IZI** | 多头查询平台 | 身份真实性、手机号实名、单日查询记录、机构类型（A/B）、事件关联（额度测算/打款/还款前后） | 2,109个 |
| **ADVANCE** | 替代数据（第三方） | 设备信息、网络行为、ocrNameMatchScore、id多头（gdYangQianGuan系列） | 265个 |
| **APP安装** | 用户手机App列表 | 安装时间间隔、App分类占比、特定App（WhatsApp/Shopee）、Embedding向量、TF-IDF | 4,879个 |
| **鉴权VO** | 用户填报信息+平台验证 | 行业/职位OneHot、人口统计、借贷行为、工作收入、设备平台、时间行为、联系人关系（296维）、绑卡验证 | 469个（粗筛后） |

> [!info] 同盾（TD）
> 同盾为第7个数据源，提供电信多头信息，V13特征量级为644个，归入联合建模范畴。

### 各数据源特征维度详解

**CBI 央行征信**（15大类，5,240个特征）：时间窗口（1-3月/3-12月/12月+）x 授信维度（债务人状态/产品类型/利率）x 还款行为（逾期天数/金额/频率）x 机构维度（银行/融资公司/风险等级/规模）。

> [!tip] CBI关键特征
> 一次失联融合模型中，CBI评分（`min_score_wT6M`、`mean_score_wT3M`）是最重要的两个纯特征，重要性分别为2.17%和2.11%。

**AFPI 金融科技**（12大类，5,873个特征）：时间窗口（15/30/90/180/360天）x 借款金额等级 x 期限 x 订单状态（在贷/结清/逾期）x 机构（头部/AFDC/尾部）。

> [!tip] AFPI关键特征
> 双塔模型中，AFPI借款信息（`avgBalance` / `maxBalance` / `minAmount`）是最重要的特征类别，重要性约5.9%。

**IZI 多头查询**（11大类，2,109个特征）：时间窗口（1天到90天+）x 查询主体（身份证/手机号/电话）x 机构类型 x 事件关联（额度测算/打款/还款前后）。

**APP安装**（6大类，4,879个特征）：时间间隔、分类统计、Embedding向量、特定App（WhatsApp/Shopee）、基础统计、TF-IDF。

**鉴权VO**（10+大类，469个粗筛后特征）：行业/职位OneHot、人口统计（28维）、借贷行为（32维）、工作收入（6维）、设备平台（25维）、时间行为（28维）、**联系人关系（296维）**、绑卡验证（12维）。

> [!info] 鉴权特征挖掘方向（已识别待开发）
> - 信息一致性特征：年龄-工龄逻辑校验、教育-职业匹配度、收入-职业/年龄合理性
> - 紧急联系人的征信水平、征信分等
> - 鉴权借款信息 x CBI/AFPI 交叉验证

### 埋点数据

埋点数据是内部行为数据源（非外部征信），核心用途是**检测代办中介（远程控制）和自动化欺诈**：

| 行为信号 | 风控含义 |
|---------|---------|
| 频繁切出/切入App | 中介同时操控多台设备 |
| 短时异常高频操作（10s/1min/5min桶） | 机器人或脚本行为 |
| 截屏事件 | 中介截屏保存用户信息用于后续套利 |
| 夜间操作比例高 | 中介集中晚上批量操作 |
| 频繁IP/设备/网络切换 | VPN/代理/模拟器/设备农场 |
