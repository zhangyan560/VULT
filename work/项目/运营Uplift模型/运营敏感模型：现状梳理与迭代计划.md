## **一、背景与目的**

  

### **背景**

  

- **过去**：依赖人工找「节点」（某类行为发生后做实验），样本分散、复用难。

- **转向目标**：模型驱动圈人——先以**日批离线大盘**为主，再演进到**实时事件触发**，并持续用效果数据自动评估与迭代。

- **约束条件**：WA 触达、合规与频控、放款增量等业务目标需可度量、可执行。

  

### **目的**

  

- 减少人工选节点，把「谁值得今天触达、触达强度」交给**模型 + 规则引擎**。

- 分阶段落地：先离线日批跑通闭环，再补实时，再做动态评估与规则/模型迭代。
    

- **可解释、可审计**：模型输出能映射到「发不发、发几条、何时发」。

## **三、各阶段详细说明**

  

### **阶段1：每天早上扫大盘离线数据**

  

|   |   |
|---|---|
|维度|内容|

| **需要什么模型** | **日批排序/分档模型**：输出「今日是否值得触达」及信号强度（连续分或分桶）。理想方向是 **uplift / 因果增量**（触达相对不触达的额外放款）；MVP 可用校准后的响应概率 + 对照/探索流量缓解偏差。 |

| **根据模型输出做什么** | ① 生成今日触达名单；<br>② 按强度映射规则决定 WA 条数（弱/中/强）；<br>③ 执行全局与用户级频控；<br>④ 全量打标模型版本、分档、名单批次，便于回溯。 |

  

> **小结**：模型每天给「人 + 强度」，业务侧负责下发 WA 和硬规则兜底；在这一阶段，还没有纳入用户当天的实时信号，直接在离线基础上让用户再过实时的触达节点。

  

---

  

### **阶段2：纳入当天实时信息，模型实时触发圈人**

  

|   |   |
|---|---|
|维度|内容|

| **需要什么模型** | **低延迟决策**：常见组合是「日批底分 + 实时特征修正」或轻量在线模型（蒸馏/小模型），在事件发生后秒级～分钟级给出是否触达及优先级；与阶段1共用同一套目标与标签定义。 |

| **根据模型输出做什么** | ① 事件进**决策服务**（统一入口）；<br>② 实时结果与当日已触达状态合并，避免超频；<br>③ 与阶段1名单去重、优先级规则写死（如实时高意向可插队或仅补充）；<br>④ SLA 与**降级**（模型/特征故障时只跑规则或停发）。 |

  

> **小结**：事件驱动时，模型给瞬时圈人信号，用实时引擎执行 WA，并与日批策略同一套频控与目标。

  

---

  

### **阶段3：动态化评估效果，迭代学习圈人规则**

  

|   |   |
|---|---|
|维度|内容|

| **需要什么模型** | 一套闭环：**监测与评估层**（分层增量、校准、漂移、实验显著性）+ **策略层**（阈值、分档、配额、「强度→条数」映射）的自动或半自动更新；模型侧定期重训/增量学习；规则侧可用 **bandit / 约束优化** 在探索与利用间分配流量。 |

| **根据模型输出做什么** | ① 每日/每周看**评估看板**（含负向指标）；<br>② 对系统给出的规则变更建议做**签核或自动生效**（按风险分级）；<br>③ 保留小比例**探索流量**，避免模型越学越偏；<br>④ 差策略自动降权或下线。 |

  

> **小结**：模型与规则在统一指标下自评、自调，人主要负责**红线、签核与异常熔断**。

## **八、Uplift 敏感模型构建思路**

  

### **8.1 核心目标定义**

  

Uplift 模型要回答的问题：**「发 WA 相对不发 WA，能额外带来多少放款？」**

  

```Plain
Uplift(user) = P(放款 | 发WA, user特征) - P(放款 | 不发WA, user特征)
```

  

- Uplift > 0：说服型用户（Persuadables）—— 重点触达
    
- Uplift ≈ 0：无论发不发都会/不会放款 —— 节省配额
    
- Uplift < 0：反效果用户（Do-Not-Disturbs）—— 主动屏蔽
    

---

  

### **8.2 样本设计**

  

```Plain
样本来源
├── 正样本：有长期对照实验的场景（实验组 vs 对照组）
│   ├── 实验组：发送了WA消息的用户
│   └── 对照组：未发送WA消息的用户（真实反事实）
│
├── Label定义
│   ├── 主要Label：观测窗口内是否发生放款（7天/14天/30天）
│   └── 辅助Label：WA点击、申请提交、放款金额（多目标扩展）
│
├── 样本窗口
│   ├── 特征窗口：触达事件发生前 T-30 ～ T-1 天的用户状态
│   └── 观测窗口：触达事件后 T+1 ～ T+N 天的结果
│
└── 样本偏差处理
    ├── 当前无对照组场景：使用 PSM（倾向得分匹配）构造近似对照
    ├── 跨场景样本合并：统一定义"动作=发WA"，区分场景作为特征输入
    └── 保留小比例探索流量作为持续学习语料
```

  

---

  

### **8.3 特征工程**

  

|   |   |   |
|---|---|---|
|特征类别|具体特征|说明|

| **用户基础** | 信用评级(A1-H)、KYC状态(KTP)、注册时长、历史借款次数 | 稳定特征，离线计算 |

| **贷款状态** | 当前可借额度、在贷/结清状态、历史逾期次数、最近还款时间 | T-1 离线 |

| **行为活跃度** | 7/14/30天登录频次、App打开时间段偏好、功能浏览路径 | 滑动窗口统计 |

| **历史触达响应** | 历史WA接收次数、点击率、历史实验组归属 | 防止偏差积累 |

| **产品偏好** | 曾申请的产品类型、利率敏感度（日利率>0.2%分层）、券使用记录 | 与当前策略对齐 |

| **实时信号（阶段2）** | 当日登录事件、还款事件、风控跑批结果 | 在线特征，秒级更新 |

| **触达动作** | 消息类型(utility/marketing/chatbot)、发送时间、发送条数 | 作为treatment变量 |

  

---

  

### **8.4 模型选型与分阶段落地**

  

```Plain
阶段1 MVP：Two-Model（T-Learner）
├── 模型A：在实验组样本上训练 P(放款 | 发WA, X)
├── 模型B：在对照组样本上训练 P(放款 | 不发WA, X)
└── Uplift分 = 模型A预测值 - 模型B预测值
    优点：简单易上线，快速验证
    缺点：两个模型分布差异大时噪声高

阶段2 提升：X-Learner / Causal Forest
├── X-Learner：交叉学习，适合实验组/对照组不均衡的情况
├── Causal Forest：非参数方法，捕捉异质性处理效应（HTE）
└── 适合特征维度高、用户分层复杂的场景

阶段3 在线化：轻量蒸馏模型
├── 将离线 Causal Forest 蒸馏为 GBDT 或浅层神经网络
├── 支持秒级推断，接入实时决策服务
└── 与阶段1日批底分结合（底分+实时修正分）
```

```Plain
│ 完件T0-10min │ 首贷已完件               │ utility / marketing / push  │
│ 还款后T2H    │ 复贷结清-触发            │ WA图文实验                  │
│ 打款后T+1天  │ 复贷在贷-续借月维度       │ chatbot开场白               │
│ 跑批完成T2H  │ 复贷续借-额度更新后       │ utility手机号               │
│ 登录事件T0   │ 复贷-头中部无offer        │ 返现/降息消息               │
│ 发券后       │ 复贷-查看还款计划         │ marketing消息               │
└──────────────┴──────────────────────────┴─────────────────────────────┘

Uplift模型在每个节点为每个用户输出：
  - 是否触达（0/1）
  - 触达强度（弱/中/强 → 对应WA条数）
  - 最优动作（消息类型选择）
```

## **九、多处理组 Uplift 模型设计：面向复杂运营场景**

  

### **9.1 问题的本质差异**

  

传统 uplift 是一个**二元因果推断**问题：

  

```Plain
传统场景（简单）：
  Control：千3费率
  Treatment：千2费率（降费）
  目标：找到对降费最敏感的用户
  Treatment 空间大小 = 1
```

  

本场景是一个**多处理组 + 多维度**问题：

  

```Plain
本场景（复杂）：
  Treatment 空间 = 场景(4) × 节点(N) × 消息类型(3~4) × 强度(弱/中/强)
  ≈ 数十到上百个 treatment 组合
  且：
  - 各组实验独立设计，样本来源、时间窗口、人群定义不统一
  - 同一用户可能同时命中多个 treatment
  - 各 treatment 之间存在干扰（spillover）
```

  

**不能直接套用传统 uplift 的原因：**

- 无法为每个 treatment 组合单独建一个 uplift 模型（样本量不足、维护成本爆炸）
    
- 不同实验的对照组定义不统一，"不发WA"的含义在各场景下不同
    
- treatment 之间有优先级与互斥关系（同一天不能同时发多个场景的消息）
    

---

  

### **9.2 设计原则：解耦 treatment 空间**

  

**核心思路**：不把 treatment 当作模型的"标签类别"，而是把它**拆解为特征维度**，用统一的样本池 + 统一的 uplift 框架建模。

  

```Plain
treatment 空间解耦
│
├── 维度1：是否触达（触达 vs 完全不触达）       ← 最核心的 binary uplift
│         这一步解决"值不值得发"
│
├── 维度2：触达时机（哪个节点触发）              ← 时间/事件特征化
│         T0/T1/T2H → 转化为"距离事件N小时"特征
│
├── 维度3：触达场景（用户处于哪个生命周期）      ← 人群状态特征化
│         首贷未完件/复贷在贷 → 转化为用户状态特征
│
└── 维度4：消息类型（发哪种消息）                ← 条件最优动作选择
          utility/marketing/chatbot → 第二阶段策略优化
```

  

**关键洞见**：

- 维度1（触达 vs 不触达）是 uplift 模型的核心，用统一的全局样本池建模
    

- 维度2/3 是**特征**，不是**treatment**，编码进模型输入

- 维度4 是在"确认触达"后的**条件策略选择**，单独处理

  

---

  

### **9.3 两阶段建模架构**

  

```Plain
┌───────────────────────────────────────────────────────────────────┐
│  第一阶段：全局触达 Uplift 模型（触达 vs 完全不触达）               │
│                                                                   │
│  输入：用户特征 X + 场景特征 S + 节点特征 T                        │
│  输出：uplift分 = P(放款|发WA) - P(放款|不发WA)                   │
│  决策：uplift > 阈值 → 进入触达名单                                │
└───────────────────────────────────────────────────────────────────┘
                              │ 确认触达
                              ▼
┌───────────────────────────────────────────────────────────────────┐
│  第二阶段：消息类型最优动作选择（Multi-arm 策略模型）               │
│                                                                   │
│  输入：用户特征 X + 场景 S                                         │
│  候选动作：{utility, marketing, chatbot, push}                    │
│  输出：各动作的期望增量，选最优                                     │
│  方法：Multi-treatment uplift 或 Contextual Bandit                │
└───────────────────────────────────────────────────────────────────┘
```

  

---

  

### **9.4 第一阶段：统一样本池构建**

  

**目标**：建立一个跨场景、跨节点的统一训练样本，使模型能学到"在任意场景/节点下，触达的边际效益"。

  

#### **Step 1：定义全局 Treatment 与 Control**

  

```Plain
Treatment（实验组）：
  凡是被发送了任意一条 WA 的用户（覆盖所有12个场景）
  记录：用户ID + 触达时间 + 场景 + 节点 + 消息类型

Control（对照组）：来源有两种
  ① 各场景实验中的"空白组/对照组"用户     ← 优先使用，质量最高
  ② 建立一个全局 Holdout Group（5~10%）   ← 永远不触达，作为黄金对照
```

  

#### **Step 2：解决不同实验的样本不可比问题**

  

各场景的实验组/对照组由于**入组规则不同**，直接合并会引入选择偏差。解决方案：

  

```Plain
方法：倾向得分加权（IPW，Inverse Probability Weighting）

1. 训练一个"倾向模型"：P(被触达 | 用户特征X, 场景S, 节点T)
   → 估计每个用户在当前场景/节点下被触达的概率

2. 用倾向得分对样本加权：
   Treatment 样本权重 = 1 / P(触达|X,S,T)
   Control  样本权重 = 1 / (1 - P(触达|X,S,T))

3. 加权后，treatment 组和 control 组的协变量分布趋于一致
   → 可以跨场景合并建模
```

  

#### **Step 3：特征编码——把场景/节点转化为特征**

  

```Plain
用户特征 X（已有）：
  信用评级、可借额度、活跃度、历史响应率 ...

场景特征 S（新增编码）：
  - 贷款生命周期阶段：[首贷未完件=1, 首贷已完件=2, 复贷结清=3, 复贷在贷=4]
  - 是否KTP：[0/1]
  - 客群价值分层：[头部=3, 中部=2, 尾部=1]

节点特征 T（新增编码）：
  - 触发事件类型：[注册后=1, 完件后=2, 还款后=3, 打款后=4, 登录后=5, 跑批后=6]
  - 距离事件发生的时间：[T+10min, T+1天, T+2天, ...]
  - 距离上次触达的时间间隔
  - 当日是否已被其他场景触达（防干扰特征）
```

  

#### **Step 4：统一 Label 定义**

  

```Plain
主要 Label（7天放款）：
  观测窗口内（触达后7天）是否发生放款 → 0/1
 ！ 这里有个问题，同一用户会不会触达一次7天内又被触达，这次放款属于哪个触达的贡献。放款归因于最近一次触达。
辅助 Label（多目标）：
  - 14天放款（更长窗口）
  - 放款金额（回归任务）
  - WA点击（近端指标，用于数据稀少时的代理指标）

注意：
  Label 打标以"首次触达时间"为起点，
  避免同一用户多次被触达导致的 label 污染
```

  

---

  

### **9.5 第二阶段：多消息类型最优动作选择**

  

在确认用户值得触达后，需要决定发哪种消息。这是一个 **多臂问题（Multi-arm）**。

  

#### **方案A：Multi-treatment T-Learner（样本充足时）**

  

```Plain
对每种消息类型训练一个响应模型：
  模型_utility    ：P(放款 | 发utility,   用户特征X)
  模型_marketing  ：P(放款 | 发marketing, 用户特征X)
  模型_chatbot    ：P(放款 | 发chatbot,   用户特征X)

各模型均以"空白组（不发WA）"为 control 基准，
Uplift_k(user) = P(放款 | 消息类型k, X) - P(放款 | 不发WA, X)

选择 Uplift 最大的消息类型作为最优动作
```

  

#### **方案B：Contextual Bandit（样本不足或探索阶段）**

  

```Plain
当某些消息类型历史样本稀少时，使用 Bandit 平衡探索与利用：

LinUCB / Thompson Sampling：
  每次决策时，对每个消息类型估计：
  期望收益 = 预测均值 + 探索奖励（不确定性越高，奖励越大）

  → 对历史数据少的类型给更多探索机会
  → 随着数据积累，自动收敛到最优消息类型

适用场景：
  新上线的消息类型（如新图文模板刚上线）
  小众用户群（样本稀少，置信区间宽）
```

  

#### **方案C：统一 Treatment Embedding（推荐长期方向）**

  

```Plain
将 treatment 类型做 embedding，与用户特征一起输入模型：

输入层：
  [用户特征 X] + [treatment embedding(消息类型)]

输出层：
  P(放款 | X, treatment)

优势：
  - 利用不同 treatment 之间的相关性（新图旧文 和 新图新文 共享图片特征）
  - 训练数据跨 treatment 共享，解决单个 treatment 样本不足问题
  - 可以泛化到新 treatment（zero-shot 预测新消息类型的效果）
```

  

---

  

### **9.6 样本污染与干扰的处理**

  

**挑战**：同一用户在同一天可能被多个场景的实验同时命中，导致 treatment 和 control 不纯。

  

```Plain
处理策略：

1. 用户级实验隔离（推荐）
   ├── 在全局实验层，将用户随机分入"WA实验池"和"全局Holdout"
   ├── 全局Holdout用户：永远不发任何场景的WA → 干净的全局对照组
   └── WA实验池用户：按各场景规则正常触达 → 实验组

2. 触达窗口隔离
   ├── 同一用户触达后，开启N天"静默窗口"，此窗口内不计入其他场景的样本
   └── 避免用户在连续被多次触达时，label被污染

3. 特征记录当日触达历史
   └── 在特征中加入"今日已触达次数"、"今日已触达场景列表"
       让模型自己学习多次触达的干扰效应
```

但是无法改变现有的实验结果

**---**

  

### **9.7 全局样本设计示意**

  

```Plain
全体用户池
│
├── 全局 Holdout（5~10%）─────────────────────────────────────────────►  永远不发WA
│   （每个用户随机分配，长期保持）                                         作为黄金对照基准
│
└── 运营实验池（90~95%）
    │
    ├── 场景A：首贷未完件
    │   ├── 实验组：发WA图文
    │   └── 空白组：不发（但可以收到其他场景WA）
    │
    ├── 场景B：复贷续借-跑批T2H
    │   ├── 实验组1：utility消息
    │   ├── 实验组2：marketing消息
    │   └── 空白组：不发
    │
    └── ...（其他场景）

建模样本逻辑：
  ┌────────────────────────────────────────────────────────────────┐
  │ Treatment = 任意场景实验组用户（加IPW权重校正跨场景分布差异）   │
  │ Control   = 全局Holdout用户（最干净）+各场景空白组（次优）      │
  │ 特征      = 用户特征 + 场景特征 + 节点特征（统一编码）          │
  │ Label     = 触达后7天内是否放款                                 │
  └────────────────────────────────────────────────────────────────┘
```

|   |   |   |   |
|---|---|---|---|
|阶段|模型形态|核心工作|解决的问题|

| **MVP** | 单场景二元 Uplift | 选取样本最多的1个场景（复贷续借跑批T2H），建经典 T-Learner | 跑通建模流程，验证增量效果 |

| **扩展** | 跨场景统一 Uplift | 用 IPW 合并多场景样本，场景/节点特征化，建统一打分模型 | 解决样本不足、跨场景泛化 |

| **优化** | 两阶段模型 | 触达决策模型 + 消息类型 Bandit | 同时优化"发不发"和"发什么" |

| **成熟** | Treatment Embedding | 统一 embedding 模型，支持新 treatment 泛化 | 新消息类型冷启动问题 |

## **十、Uplift 模型 + Agent 自动化运营体系**

  

### **10.1 整体架构设想**

  

```Plain
                    ┌─────────────────────────────────────┐
                    │          自动化运营 Agent 系统        │
                    └─────────────────────────────────────┘
                                      │
          ┌───────────────────────────┼───────────────────────────┐
          │                           │                           │
          ▼                           ▼                           ▼
┌──────────────────┐       ┌──────────────────┐       ┌──────────────────┐
│   感知层 Agent    │       │   决策层 Agent    │       │   执行层 Agent    │
│ (数据 & 特征)     │       │ (Uplift 模型)     │       │ (WA 发送 & 频控) │
└──────────────────┘       └──────────────────┘       └──────────────────┘
          │                           │                           │
          ▼                           ▼                           ▼
┌──────────────────┐       ┌──────────────────┐       ┌──────────────────┐
│   监控层 Agent    │◄─────►│   学习层 Agent    │◄─────►│   治理层 Agent    │
│ (效果评估)        │       │ (模型迭代)        │       │ (规则 & 熔断)     │
└──────────────────┘       └──────────────────┘       └──────────────────┘
```

  

---

  

### **10.2 各 Agent 职责**

  

**感知层 Agent（Perception Agent）**

```Plain
职责：
├── 实时监听用户事件流（还款/登录/风控通过/发券）
├── 聚合离线特征（T-1批次）与在线特征（实时信号）
├── 触发决策层（判断是否进入打分流程）
└── 输出：用户特征向量 + 触发事件类型
```

  

**决策层 Agent（Decision Agent）**

```Plain
职责：
├── 调用 Uplift 模型，输出每个用户的增量得分
├── 结合频控状态，决定是否触达（pass/block）
├── 根据得分分档，选择触达强度（发1/2/3条）
├── 根据用户特征，选择最优消息类型（utility/marketing/chatbot）
└── 输出：触达名单 + 消息类型 + 发送时间建议
```

  

**执行层 Agent（Execution Agent）**

```Plain
职责：
├── 接收决策层输出，执行 WA 发送
├── 全局频控检查（用户级/渠道级/配额级）
├── 消息下发、回执记录、日志落库
└── 降级处理（模型异常时切换规则兜底）
```

  

**监控层 Agent（Monitoring Agent）**

```Plain
职责：
├── 实时监控发送成功率、点击率、转化率
├── 计算分层增量效果（实验组 vs 对照组）
├── 检测模型分数漂移和群体分布变化（PSI）
├── 发现异常自动触发熔断或告警
└── 输出：效果报告 + 异常事件队列
```

  

**学习层 Agent（Learning Agent）**

```Plain
职责：
├── 收集效果数据，构造新一轮训练样本
├── 触发模型重训（定期 or 漂移超阈值）
├── 运行 Bandit 算法，动态调整探索/利用比例
├── 输出新模型版本，提交治理层审核
└── 维护长期反转实验，防止模型偏移
```

  

**治理层 Agent（Governance Agent）**

```Plain
职责：
├── 模型变更签核（低风险自动生效，高风险人工审批）
├── 维护业务红线（频控上限、合规规则、负向指标阈值）
├── 异常熔断（自动降级或停发）
└── 版本管理与回滚
```

### **10.3 实现方案（分阶段落地）**

  

|   |   |   |   |
|---|---|---|---|
|阶段|目标|关键交付|依赖条件|

| **Phase 0（现在）** | 打通数据与实验 | 配置长期对照实验；统一日志格式 | 各场景补配对照组 |

| **Phase 1（3个月）** | 离线 Uplift 模型上线 | T-Learner 模型跑批；日批名单替代人工选节点 | 足量实验样本（≥3个月） |

| **Phase 2（6个月）** | 决策层 Agent 接入 | 打分服务化；频控统一入口；消息类型自动选择 | 在线特征工程基础设施 |

| **Phase 3（9个月）** | 实时事件触发 | 感知层+执行层 Agent；秒级响应还款/登录事件 | 实时消息队列（Kafka/Flink） |

| **Phase 4（12个月）** | 全闭环自动化 | 监控+学习+治理 Agent 联动；模型自迭代 | 评估体系稳定；签核流程制度化 |

### **10.4 潜在难点**

  

**数据层难点**

- 当前大多数场景无对照组，**样本质量是最大瓶颈**，需先补实验再建模

- 用户在多个场景间的串扰（spillover effect）导致因果估计偏差
    
- 实时特征工程复杂度高，需要专门的特征平台支撑
    

**模型层难点**

- Uplift 模型评估比普通分类模型难，Qini/AUUC 指标不直观，需建立业务对照
    
- 用户行为漂移（节假日、产品变化）会导致模型快速失效，需高频重训
    
- 多目标权衡（放款增量 vs 用户体验 vs 不良率）难以用单一指标优化
    

**工程层难点**

- 多 Agent 协作的一致性问题：感知、决策、执行之间的状态同步
    
- 降级策略设计：模型/特征任一故障时系统需无缝切换到规则兜底
    
- 实时决策服务的 SLA 要求（<100ms），对模型推断效率要求高
    

**治理层难点**

- 模型自动更新与业务签核之间的节奏不匹配
    
- 长期探索流量的设计：比例太小无法学习，比例太大影响业务效果
    
- 合规风险：WA 触达频次、内容合规需要持续人工监督

|   |   |
|---|---|
|价值维度|具体收益|

| **成本节省** | 减少无效 WA 发送量（WA 按条计费），降低渠道成本；减少用户退订，保护长期触达能力 |

| **风险管控** | 自动检测模型漂移、异常熔断，减少因策略失效导致的业务损失 |

| **可解释性** | Uplift 分档可映射到「发不发、发几条、何时发」，满足合规审计要求 |
