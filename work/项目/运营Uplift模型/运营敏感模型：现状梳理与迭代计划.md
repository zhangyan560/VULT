## **一、背景与目的**

  

### **背景**

  

- **过去**：依赖人工找「节点」（某类行为发生后做实验），样本分散、复用难。

- **转向目标**：模型驱动圈人——先以**日批离线大盘**为主，再演进到**实时事件触发**，并持续用效果数据自动评估与迭代。

- **约束条件**：WA 触达、合规与频控、放款增量等业务目标需可度量、可执行。

  

### **目的**

  

- 减少人工选节点，把「谁值得今天触达、触达强度」交给**模型 + 规则引擎**。

- 分阶段落地：先离线日批跑通闭环，再补实时，再做动态评估与规则/模型迭代。
    

- **可解释、可审计**：模型输出能映射到「发不发、发几条、何时发」。

## **三、各阶段详细说明**

  

### **阶段1：每天早上扫大盘离线数据**

  

|   |   |
|---|---|
|维度|内容|

| **需要什么模型** | **日批排序/分档模型**：输出「今日是否值得触达」及信号强度（连续分或分桶）。理想方向是 **uplift / 因果增量**（触达相对不触达的额外放款）；MVP 可用校准后的响应概率 + 对照/探索流量缓解偏差。 |

| **根据模型输出做什么** | ① 生成今日触达名单；<br>② 按强度映射规则决定 WA 条数（弱/中/强）；<br>③ 执行全局与用户级频控；<br>④ 全量打标模型版本、分档、名单批次，便于回溯。 |

  

> **小结**：模型每天给「人 + 强度」，业务侧负责下发 WA 和硬规则兜底；在这一阶段，还没有纳入用户当天的实时信号，直接在离线基础上让用户再过实时的触达节点。

  

---

  

### **阶段2：纳入当天实时信息，模型实时触发圈人**

  

|   |   |
|---|---|
|维度|内容|

| **需要什么模型** | **低延迟决策**：常见组合是「日批底分 + 实时特征修正」或轻量在线模型（蒸馏/小模型），在事件发生后秒级～分钟级给出是否触达及优先级；与阶段1共用同一套目标与标签定义。 |

| **根据模型输出做什么** | ① 事件进**决策服务**（统一入口）；<br>② 实时结果与当日已触达状态合并，避免超频；<br>③ 与阶段1名单去重、优先级规则写死（如实时高意向可插队或仅补充）；<br>④ SLA 与**降级**（模型/特征故障时只跑规则或停发）。 |

  

> **小结**：事件驱动时，模型给瞬时圈人信号，用实时引擎执行 WA，并与日批策略同一套频控与目标。

  

---

  

### **阶段3：动态化评估效果，迭代学习圈人规则**

  

|   |   |
|---|---|
|维度|内容|

| **需要什么模型** | 一套闭环：**监测与评估层**（分层增量、校准、漂移、实验显著性）+ **策略层**（阈值、分档、配额、「强度→条数」映射）的自动或半自动更新；模型侧定期重训/增量学习；规则侧可用 **bandit / 约束优化** 在探索与利用间分配流量。 |

| **根据模型输出做什么** | ① 每日/每周看**评估看板**（含负向指标）；<br>② 对系统给出的规则变更建议做**签核或自动生效**（按风险分级）；<br>③ 保留小比例**探索流量**，避免模型越学越偏；<br>④ 差策略自动降权或下线。 |

  

> **小结**：模型与规则在统一指标下自评、自调，人主要负责**红线、签核与异常熔断**。

## **九、新干净实验设计**



### **9.1 三组实验设计**



**核心原则**：三组完全互斥，对照组不参与任何触达决策，保证因果纯洁性。



```
实验流量切分（基于 user_id 哈希）：
├── 对照组（15%）：不触达，仅记录自然放款行为
├── 跑批触达组（15%）：跑批 uplift 分 ≥ 阈值则触达
└── 联合触达组（10%）：跑批分 ≥ 阈值 AND 实时信号触发才触达
```



### **9.2 Y 变量定义**



| 分组 | Y 计算起点 | 观测窗口 |
|---|---|---|
| 对照组 | stat_date | T+3 |
| 跑批触达组 | 消息发送时间 | T+3 |
| 联合触达组 | 实时事件触发时间 | T+3 |



- **跑批模型 uplift**：Y = 触达后 T+3 内是否放款（0/1）
- **实时模型 uplift**：Y = 实时事件（登录/还款）发生后 T+3 内是否放款（0/1）



**注意**：对照组的 Y 采用同 stat_date 对齐，统一与实验组比较同时间段的自然放款率。



### **9.3 污染控制**



- 同一用户在实验周期内只属于一个分组（哈希固定）
- 跑批触达组和联合触达组共用同一套跑批模型，区别只在实时层
- 联合触达组实时信号仅作为触发条件，不影响跑批打分逻辑
- 实验期间不对对照组做任何渠道触达（包括非 WA 渠道）

### **10.1 数据来源与样本设计**

- **特征窗口**：stat_date 前 T-30 ～ T-1 天的用户状态特征

| 最小样本量 | 单组 ≥ 5 万有效用户（触达组 + 对照组各自） |



### **10.2 建模方法**



**方法一：T-Learner（Two-Model）**



```python
# 分别在实验组/对照组训练两个 LGB 模型
model_t = lgb.train(params, train_data[train_data.treatment==1])
model_c = lgb.train(params, train_data[train_data.treatment==0])

# uplift 得分 = 实验组预测 - 对照组预测
uplift = model_t.predict(X) - model_c.predict(X)
```



**方法二：X-Learner（跨模型残差修正）**

- Stage 1：同 T-Learner，训练 τ̂_t 和 τ̂_c
- Stage 2：用对照组模型预测实验组的反事实，计算残差，反之亦然
- Stage 3：用倾向得分加权合并两组残差估计



**特征筛选流程**：LGB 特征重要性（weight + total_gain）→ 递归消融实验 → 最终特征数 < 1000



### **10.3 评估指标**



- **AUUC**（Area Under Uplift Curve）：使用 `causalml.metrics.auuc_score`
- **Qini 曲线**：等切 10 bin，计算每 bin 内实验组 vs 对照组的增量放款率
- **OOT 稳定性**：Train AUUC vs OOT AUUC 不超过 5%

## **十一、实时模型建模方法**



### **11.1 实时触达场景定义**



实时模型解决的问题：**用户在 App 内发生特定行为（登录、还款、浏览）后，是否应该立即触达？触达的增量收益是多少？**



两种核心触发事件：



| 事件类型 | 触发条件 | 典型场景 |
|---|---|---|
| 登录事件 | 用户打开 App 或主动登录 | 唤醒沉默用户，展示个性化额度 |
| 还款事件 | 用户成功完成还款 | 还款后立即推荐复贷 |



### **11.2 Y 变量重定义**



- Y 的计算起点不再是 stat_date，而是**事件发生时刻 t_event**
- 观测窗口：事件后 T+3 天内是否放款（0/1）
- 对照组：发生同类事件但未被触达的用户，Y 同口径统计



```
实时 uplift(user) = P(放款 | 事件后触达) - P(放款 | 事件后未触达)
```



### **11.3 实时特征工程**



| 特征类型 | 内容 | 更新频率 |
|---|---|---|
| 事件特征 | 本次登录时间、距上次登录间隔、本次还款金额 | 实时（事件触发时计算） |
| 近期行为窗口 | 近 7 天登录次数、近 3 次还款金额趋势 | 小时级 |
| 贷款状态 | 当前可借额度、在贷金额、账期状态 | T-1 离线 |
| 跑批 uplift 分 | 昨日跑批模型输出的 uplift 得分 | 日级 |
| 用户基础特征 | 信用评级、KYC、注册时长 | 日级 |



### **11.4 建模方案**



- **短期方案（阶段1）**：以新干净实验的联合触达组数据，用 T-Learner 训练离线实时 uplift 模型；事件触发时查询模型输出，分高于阈值则触达
- **中期方案（阶段2）**：接入实时流特征（Flink/Kafka），实现事件触发时毫秒级打分；使用轻量级在线模型（LR / 小规模 GBDT）部署
- **长期方案（阶段3）**：跑批模型（全局 uplift）+ 实时模型（行为触发 uplift）联合决策，用 Bandit 动态调整两者权重



### **11.5 实验验证逻辑**



新干净实验中，联合触达组与跑批触达组的差值即为实时模型的净增益：



```
实时增益 = ATE(联合触达组) - ATE(跑批触达组)
```



ATE 估计基于对照组基准放款率，三组实验支持干净地拆解跑批贡献与实时贡献。

## **十二、Uplift 模型 + Agent 自动化运营体系**

  

### **10.1 整体架构设想**

  

```Plain
                    ┌─────────────────────────────────────┐
                    │          自动化运营 Agent 系统        │
                    └─────────────────────────────────────┘
                                      │
          ┌───────────────────────────┼───────────────────────────┐
          │                           │                           │
          ▼                           ▼                           ▼
┌──────────────────┐       ┌──────────────────┐       ┌──────────────────┐
│   感知层 Agent    │       │   决策层 Agent    │       │   执行层 Agent    │
│ (数据 & 特征)     │       │ (Uplift 模型)     │       │ (WA 发送 & 频控) │
└──────────────────┘       └──────────────────┘       └──────────────────┘
          │                           │                           │
          ▼                           ▼                           ▼
┌──────────────────┐       ┌──────────────────┐       ┌──────────────────┐
│   监控层 Agent    │◄─────►│   学习层 Agent    │◄─────►│   治理层 Agent    │
│ (效果评估)        │       │ (模型迭代)        │       │ (规则 & 熔断)     │
└──────────────────┘       └──────────────────┘       └──────────────────┘
```

### **10.2 各 Agent 职责**

  

**感知层 Agent（Perception Agent）**

```Plain
职责：
├── 实时监听用户事件流（还款/登录/风控通过/发券）
├── 聚合离线特征（T-1批次）与在线特征（实时信号）
├── 触发决策层（判断是否进入打分流程）
└── 输出：用户特征向量 + 触发事件类型
```

  

**决策层 Agent（Decision Agent）**

```Plain
职责：
├── 调用 Uplift 模型，输出每个用户的增量得分
├── 结合频控状态，决定是否触达（pass/block）
├── 根据得分分档，选择触达强度（发1/2/3条）
├── 根据用户特征，选择最优消息类型（utility/marketing/chatbot）
└── 输出：触达名单 + 消息类型 + 发送时间建议
```

  

**执行层 Agent（Execution Agent）**

```Plain
职责：
├── 接收决策层输出，执行 WA 发送
├── 全局频控检查（用户级/渠道级/配额级）
├── 消息下发、回执记录、日志落库
└── 降级处理（模型异常时切换规则兜底）
```

  

**监控层 Agent（Monitoring Agent）**

```Plain
职责：
├── 实时监控发送成功率、点击率、转化率
├── 计算分层增量效果（实验组 vs 对照组）
├── 检测模型分数漂移和群体分布变化（PSI）
├── 发现异常自动触发熔断或告警
└── 输出：效果报告 + 异常事件队列
```

  

**学习层 Agent（Learning Agent）**

```Plain
职责：
├── 收集效果数据，构造新一轮训练样本
├── 触发模型重训（定期 or 漂移超阈值）
├── 运行 Bandit 算法，动态调整探索/利用比例
├── 输出新模型版本，提交治理层审核
└── 维护长期反转实验，防止模型偏移
```

  

**治理层 Agent（Governance Agent）**

```Plain
职责：
├── 模型变更签核（低风险自动生效，高风险人工审批）
├── 维护业务红线（频控上限、合规规则、负向指标阈值）
├── 异常熔断（自动降级或停发）
└── 版本管理与回滚
```

|   |   |   |   |
|---|---|---|---|
|阶段|目标|关键交付|依赖条件|

| **Phase 0（现在）** | 打通数据与实验 | 配置长期对照实验；统一日志格式 | 各场景补配对照组 |

### **10.4 潜在难点**

  

**数据层难点**

- 当前大多数场景无对照组，**样本质量是最大瓶颈**，需先补实验再建模

- 用户在多个场景间的串扰（spillover effect）导致因果估计偏差
    
- 实时特征工程复杂度高，需要专门的特征平台支撑
    

**模型层难点**

- Uplift 模型评估比普通分类模型难，Qini/AUUC 指标不直观，需建立业务对照
    
- 用户行为漂移（节假日、产品变化）会导致模型快速失效，需高频重训
    
- 多目标权衡（放款增量 vs 用户体验 vs 不良率）难以用单一指标优化
    

**工程层难点**

- 多 Agent 协作的一致性问题：感知、决策、执行之间的状态同步
    
- 降级策略设计：模型/特征任一故障时系统需无缝切换到规则兜底
    
- 实时决策服务的 SLA 要求（<100ms），对模型推断效率要求高
    

**治理层难点**

- 模型自动更新与业务签核之间的节奏不匹配
    
- 长期探索流量的设计：比例太小无法学习，比例太大影响业务效果
    
- 合规风险：WA 触达频次、内容合规需要持续人工监督

|   |   |
|---|---|
|价值维度|具体收益|

| **放款增量** | 精准识别 Persuadables，减少对 Sure Things 和 Do-Not-Disturbs 的无效触达，提升单位 WA 配额产出的放款金额 |

| **迭代速度** | 闭环自动评估+重训，从月级迭代压缩到周级甚至天级迭代 |

| **风险管控** | 自动检测模型漂移、异常熔断，减少因策略失效导致的业务损失 |

| **可解释性** | Uplift 分档可映射到「发不发、发几条、何时发」，满足合规审计要求 |
