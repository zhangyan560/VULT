# 子分模型方法论

## 一、子分模型的核心理念

### 1.1 为什么需要子分？

主模型（如一次风控MOB1模型）追求全客群的整体区分力，但在以下场景会遗漏风险信号：

- **特定数据源的深层信息**：主模型融合多源特征时，单源特征的深层组合被稀释
- **非标目标变量**：主模型Y值是MOB1逾期，但长期逾期（MOB3/MOB6）、失联、欺诈等维度需要专门建模
- **特定客群的差异化风险**：头部评级和尾部评级的风险模式不同，统一模型难以兼顾
- **时间维度的多样性**：不同历史时间段的用户行为模式可能存在差异

子分模型的本质是**用专精模型捕捉主模型遗漏的特定维度风险**，最终通过融合层将子分信息注入主模型。

### 1.2 子分与主模型的关系

```
┌─────────────────────────────────────────────────────┐
│                    融合层（LightGBM）                  │
│                                                     │
│  输入：主模型分 + 子分模型分(N个) + [可选]原始特征     │
│  输出：最终融合评分                                   │
│                                                     │
│  筛选标准：                                          │
│  - 单子分增益 > N pp（通常要求 > 0.5pp）              │
│  - OOT上稳定性（月度AUC波动小）                       │
│  - 与主模型相关性不能过高（否则无增量信息）              │
│  - 与已入选子分的冗余度低                             │
└─────────────────────────────────────────────────────┘
         ▲         ▲         ▲         ▲
         │         │         │         │
    ┌────┴───┐ ┌──┴────┐ ┌──┴────┐ ┌──┴────┐
    │ 主模型  │ │ 子分1  │ │ 子分2  │ │ 子分N  │
    │ MOB1   │ │ CBI源  │ │ 蒸馏   │ │ 流入   │
    └────────┘ └───────┘ └───────┘ └───────┘
```

## 二、12种子分设计范式

### 范式1：数据源子分

**设计思路**：对每个数据源单独建模，深度挖掘该数据源内部的特征组合。主模型融合多源时各源的权重被稀释，单源子分可以放大该数据源的区分力。

**方法**：
- 每个数据源训练独立的LightGBM模型
- 通常使用双Y值：标准Y（如MOB1）+ 辅助Y（如MOB2或LABEL180天标签）
- 辅助Y捕捉更长时间维度的风险信号

**适用数据源**：CBI征信、AFPI网贷、IZI多头、鉴权、APP安装等

**设计模板**：
```python
# 数据源子分训练模板
for source in ['CBI', 'AFPI', 'IZI', 'AUTH', 'APP', 'ADVANCE']:
    for y_label in ['mob1_label7', 'mob2_label7', 'label180']:
        model = train_lgb(
            features=source_features[source],
            label=y_label,
            params=sub_model_params
        )
        score_name = f"SUB_{source}_{y_label}"
        save_score(model, score_name)
```

### 范式2：蒸馏子分

**设计思路**：通过知识蒸馏将历史版本模型（V10/V11/V12）的预测能力迁移到当前版本的子分中。即使旧模型的特征不再使用，其学到的风险模式仍有价值。

**方法**：
1. 使用历史模型对当前训练集打分，获得soft label
2. 以soft label为目标，用当前版本的特征训练新模型
3. 新模型的输出即为蒸馏子分

```python
# 蒸馏子分流程
# Step 1: 旧模型打分
old_model_score = old_model.predict(current_features)

# Step 2: 以旧模型分为目标训练新模型（回归）
distill_model = train_lgb(
    features=current_features,
    label=old_model_score,    # soft label
    objective='regression',
    metric='mse'
)
```

**价值**：旧模型可能学到了某些在当前特征体系中不易直接表达的非线性风险模式，蒸馏保留了这种"暗知识"。

### 范式3：历史子分

**设计思路**：按时间窗口切片训练模型，捕捉不同时期的风险模式。市场环境随时间变化，不同时期的"坏用户画像"可能不同。

**方法**：
- 将训练数据按半年或季度切片
- 可选：按评级分组（如头部A-C、中部D-E、尾部F-G）
- 在每个切片 x 评级分组上训练独立模型

```
时间切片方案：
- H1: 上半年数据
- H2: 下半年数据
- 可进一步按年份组合：24H2、2425（跨年）、25H1
```

**注意**：历史子分数量容易膨胀（如74个），需严格筛选，未产生增益的应及时缩减以降低计算开销。

### 范式4：弹性模型

**设计思路**：预测短期逾期是否会演变为中期/长期逾期。核心问题是"这个用户短期逾期后还能不能还上"。

**方法**：
- MOB1to3：在MOB1已逾期的用户中，预测MOB3是否仍逾期
- MOB3to6：在MOB3已逾期的用户中，预测MOB6是否仍逾期

```python
# 弹性模型样本筛选
elastic_sample = full_sample[full_sample['mob1_label7'] == 1]  # 只选MOB1已逾期
elastic_label = elastic_sample['mob3_label7']                   # Y值：MOB3是否仍逾期
```

**业务价值**：区分"临时周转困难"和"真正还不上"的用户，帮助催收资源分配和额度调整。

### 范式5：金额加权子分

**设计思路**：标准模型中每个样本等权，但高金额样本的损失远大于低金额样本。金额加权子分对高额度样本赋予更大权重，优化高金额客群的风险排序。

**方法**：
```python
# 金额加权训练
# 权重方案：将金额离散化为权重
def amount_weight(principal, cap=1500_0000):
    """金额越高权重越大，cap封顶"""
    return min(principal / 100_0000, cap / 100_0000)

sample_weight = train['principal'].apply(lambda x: amount_weight(x, cap=1000_0000))
model = lgb.train(params, train_set, sample_weight=sample_weight)
```

**多版本对比**：通常训练多个cap值版本（如500w/1000w/1500w），选择综合增益最优的版本。

### 范式6：双塔子分

**设计思路**：借鉴推荐系统的双塔架构，将风险维度和业务维度（如下单率、额度使用率）分别建模，然后通过乘积融合。

**STAR架构公式**：

```
final_score = (risk_tower + offset) * business_tower

其中：
- risk_tower: 风险塔输出（MOB1逾期概率）
- business_tower: 业务塔输出（下单概率 / 额度使用率）
- offset: 偏移常数（通常0.03~0.1），避免risk_tower为0时乘积归零
```

**设计变体**：
- 下单率 x 风险：识别"高下单意愿且低风险"的优质用户
- 额度使用率 x 风险：识别"高额度利用且低风险"的用户

**双塔的优势**：
- 两个塔可以独立优化，业务塔和风险塔的特征空间不必重叠
- 乘积融合天然实现了"既要风险低，又要业务价值高"的目标
- 头部bin的金额逾期率更低、额度使用率更高

### 范式7：流入子分

**设计思路**：预测用户在各MOB期"首次入逾"的风险。与标准MOB1标签不同，流入标签关注"从好变坏"的转折点。

**方法**：
- MOB1D1：MOB1第1天入逾预测（首期立即违约）
- MOB3D1：MOB3第1天入逾预测（第3期突然违约）
- MOB6D1 / MOB9D1：同理

**标签定义**：
```python
# 流入标签：在该MOB期从未逾期变为逾期
inflow_label = (mob_N_status == 'overdue') & (mob_N_minus_1_status == 'current')
```

**业务价值**：流入系列能捕捉"前期表现正常但后期突然违约"的用户，这类用户是主模型最容易遗漏的。

### 范式8：头尾部拆分

**设计思路**：按主模型评级将客群拆分为头部（低风险）和尾部（高风险），分别训练专精模型。

**拆分方式**：
- 头部（如A-C评级）：短期模型 + 长期模型
- 尾部（如D-G评级）：短期模型 + 长期模型

**设计原因**：
- 头部用户的逾期率极低，需要更精细的特征组合来区分少量坏样本
- 尾部用户的风险模式更多样，需要专门的特征来捕捉不同类型的违约原因

### 范式9：Y值差异化

**设计思路**：使用非标准目标变量训练子分，捕捉主模型Y值无法覆盖的风险维度。

**常见非标Y值**：

| Y值变体 | 定义 | 捕捉维度 |
|---------|------|---------|
| 逾期90D+ | 逾期天数超过90天 | 深度逾期风险（vs 浅度逾期） |
| 逾期180D+ | 逾期天数超过180天 | 核销级风险 |
| Tweedie | 逾期金额的Tweedie分布回归 | 同时考虑逾期概率和金额 |
| 长逾30+ | 最长逾期天数超过30天 | 持续逾期行为 |
| MOB2D7 | MOB2逾期7天 | 中期违约行为 |

**Tweedie回归的优势**：

```python
# Tweedie分布同时建模"是否逾期"和"逾期多少"
params = {
    'objective': 'tweedie',
    'tweedie_variance_power': 1.5,  # 1-2之间，1.5为常用值
}
# Y值：逾期金额（大量0 + 连续正值）
```

### 范式10：反欺诈子分

**设计思路**：针对欺诈类风险专门建模，捕捉与信用风险不同的欺诈模式。

**常见反欺诈子分**：
- FPD31子分：首期逾期31天，捕捉"一开始就没打算还"的欺诈用户
- 资金饥渴子分：多平台同时大量借款行为识别
- 失联子分：预测催收时用户是否失联（MOB1逾期后7天内5天联系不上）

### 范式11：复贷子分

**设计思路**：用复贷用户的"首贷时期特征"训练子分。复贷用户有历史表现可参考，但其首贷时的特征值已经包含了早期风险信号。

**方法**：
```python
# 复贷子分：用复贷用户样本，但只使用首贷时期的特征
reloan_sample = full_sample[full_sample['is_reloan'] == 1]
reloan_features = get_first_loan_features(reloan_sample)  # 首贷时的特征快照
reloan_label = reloan_sample['mob1_label7']  # 当前复贷的逾期标签
```

**多时间窗口**：通常按1/3/6/9个月滚动训练，捕捉不同复贷间隔的风险模式。

### 范式12：业务场景子分

**设计思路**：为特定业务场景（如额度系数计算、前置风控）训练专用模型，这些场景的优化目标与主模型不完全一致。

**常见业务场景子分**：
- 前置C卡子分：免费前筛阶段的快速风险评估
- 额度双塔子分：额度定价专用的双塔模型
- 额度使用率子分：预测用户会使用多少额度
- 多头风险/资质风险子分：额度系数调整因子

## 三、子分融合策略

### 3.1 特征级融合 vs 分数级融合

| 融合方式 | 描述 | 优点 | 缺点 |
|---------|------|------|------|
| 分数级融合 | 子分输出分数作为融合层的特征 | 简单、稳定、子分可独立迭代 | 信息压缩为单一分数 |
| 特征级融合 | 子分使用的原始特征直接进入融合层 | 信息保留完整 | 维度高、过拟合风险 |
| 混合融合 | 子分分数 + 部分原始特征 | 平衡信息量和复杂度 | 需要特征选择 |

**推荐方案**：以分数级融合为主，混合融合为辅。先评估纯分数融合的增益，如果不够再加入原始特征。

### 3.2 融合层训练

```python
# 融合层训练模板
fusion_features = ['main_model_score']

# 添加通过筛选的子分分数
for sub_score in sub_model_scores:
    if sub_score.gain > threshold_pp:
        fusion_features.append(sub_score.name)

# 训练融合LightGBM
fusion_model = train_lgb(
    features=train[fusion_features],
    label=train['mob1_label7'],
    params={
        'learning_rate': 0.01125,
        'max_depth': 3,
        'num_leaves': 8,
        'min_data_in_leaf': 200,
        'max_bin': 50,
        'is_unbalance': True,
        'reg_alpha': 1,
        'reg_lambda': 2,
        'early_stopping_rounds': 500
    }
)
```

### 3.3 筛选标准

一个子分从候选到入模，需要通过以下标准：

| 筛选维度 | 标准 | 说明 |
|---------|------|------|
| 增益 | 单子分增益 > N pp（如0.5pp） | 加入融合层后的边际AUC提升 |
| 稳定性 | OOT月度AUC波动 < 阈值 | 避免仅在特定月份有效的子分 |
| 与主模型相关性 | 相关系数不过高（如 < 0.9） | 过高说明无增量信息 |
| 与已选子分冗余 | 与已入选子分相关性 < 阈值 | 避免重复计数同一信号 |
| 交叉Lift | 在主模型各bin内仍有额外区分力 | 验证子分的增量价值 |

### 3.4 增益评估维度

| 评估维度 | 指标 | 说明 |
|---------|------|------|
| 区分度提升 | AUC / KS 差值 | OOT集上对比基线模型 |
| 风险斜率 | decile bad rate slope | 分位数逾期率单调性 |
| 交叉分箱 | baseline bin x new score bin | 在基线各分箱内的额外区分能力 |
| 时间稳定性 | 月度AUC/KS | 跨时间的性能稳定性 |
| 子群体增益 | 按评级/白户/客群拆分 | 不同人群上的差异化表现 |

## 四、蒸馏子分方法

### 4.1 知识蒸馏在子分中的应用

知识蒸馏（Knowledge Distillation）原始场景是将大模型的知识迁移到小模型。在子分体系中，蒸馏的目标是：

1. **跨版本知识保留**：旧版本模型（如V10/V11/V12）可能已下线，但其学到的风险模式仍有价值
2. **跨架构知识迁移**：额度模型、反欺诈模型等不同架构模型的知识融入风控主模型

### 4.2 蒸馏流程

```
Step 1: Teacher模型打分
  将旧版本模型（teacher）对当前训练集打分，生成soft label

Step 2: Student模型训练
  以soft label为目标（回归任务），使用当前特征体系训练新模型（student）

Step 3: 蒸馏分数评估
  评估student模型分数：
  - 与teacher分数的相关性（应较高，验证知识迁移成功）
  - 与主模型的相关性（应适中，太高无增量）
  - 对真实标签的AUC（验证预测能力）

Step 4: 融合层入选
  将蒸馏分数作为子分加入融合层
```

### 4.3 多Teacher蒸馏

可以同时蒸馏多个teacher模型：

```python
teacher_models = {
    'v10': load_model('v10_mob1'),
    'v11': load_model('v11_mob1'),
    'v12': load_model('v12_mob1'),
    'edu_v4': load_model('edu_coeff_v4'),
    'reloan': load_model('reloan_model'),
}

for name, teacher in teacher_models.items():
    soft_label = teacher.predict(current_features)
    student = train_lgb(current_features, soft_label, objective='regression')
    scores[f'distill_{name}'] = student.predict(current_features)
```

## 五、双塔子分方法

### 5.1 STAR架构

STAR（Simple Two-tower Architecture for Risk）将风险评估分解为两个独立的"塔"：

**公式**：

```
final_score = (risk_tower + offset) * business_tower
```

- **risk_tower**：风险塔，输出逾期概率，目标是MOB1/FPD31等
- **business_tower**：业务塔，输出业务指标预测（如下单概率、额度使用率）
- **offset**：偏移常数（通常0.03~0.1），确保即使风险塔输出极低值，最终分数也不会归零

### 5.2 风险塔 x 业务塔的设计思路

**下单率双塔**：

```python
# 风险塔：预测逾期概率
risk_model = train_lgb(features, label='mob1_label7')
risk_score = risk_model.predict(features)

# 业务塔：预测7天内是否下单
biz_model = train_lgb(features, label='is_order_in_7d')
biz_score = biz_model.predict(features)

# STAR融合
star_score = (risk_score + 0.03) * biz_score
```

**额度使用率双塔**：

```python
# 业务塔换为额度使用率
biz_model = train_lgb(features, label='credit_utilization_rate')
```

### 5.3 offset参数选择

offset过大会削弱风险塔的区分力，过小可能导致数值不稳定：

- 推荐范围：0.03 ~ 0.1
- 选择方法：网格搜索，评估不同offset下头部bin的金额逾期率和额度使用率
- 目标：头部bin同时实现更低的逾期率和更高的业务转化

### 5.4 加权乘积变体

除了简单乘积，还可以使用加权乘积：

```
final_score = risk_score^alpha * biz_score^(1-alpha)
```

alpha控制风险和业务的权重比例，alpha=0.5时为等权。

## 六、弹性模型方法

### 6.1 MOB1to3跨期预测

弹性模型的核心问题：**短期逾期的用户，有多大概率在中期仍然逾期？**

```
全量用户
    |
    +--> MOB1未逾期 --> 无需弹性预测（大部分用户）
    |
    +--> MOB1已逾期 --> 弹性模型输入
              |
              +--> MOB3仍逾期 --> "刚性违约"（真正还不上）
              |
              +--> MOB3已还清 --> "弹性还款"（临时困难已解决）
```

### 6.2 训练方法

```python
# Step 1: 筛选MOB1已逾期的样本
elastic_sample = full_sample[full_sample['mob1_label7'] == 1]

# Step 2: Y值为MOB3是否仍逾期
elastic_label = elastic_sample['mob3_label7']

# Step 3: 特征包含所有可用特征（不仅限于MOB1时点的特征）
elastic_features = elastic_sample[feature_columns]

# Step 4: 训练
elastic_model = train_lgb(elastic_features, elastic_label)
```

### 6.3 扩展

- **MOB3to6**：在MOB3已逾期用户中预测MOB6表现
- **多级联弹性**：MOB1to3分 + MOB3to6分联合使用

### 6.4 业务应用

弹性模型的输出用于：
- 催收资源分配：对"刚性违约"用户早期介入，对"弹性还款"用户减少催收强度
- 拨备计算：更精准估计不同逾期阶段的实际损失
- 额度调整：对弹性还款概率高的用户保留额度空间
