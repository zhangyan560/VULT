# 特征挖掘 SOP（标准操作流程）

## 一、整体架构概览

整个特征挖掘体系服务于**印尼现金贷（EasyCash/Fintopia）信用风险评估**，目标变量为 `mob1_label7`（放款后1个月逾期7天+）和 `mob3_label7`（3个月逾期）。数据涵盖**8大数据源**，特征衍生采用**7大技术路线**。

```
┌─────────────────────────────────────────────────────────────┐
│                      数据源层                                │
│  CBI征信 │ AFPI网贷 │ APP安装 │ 埋点行为 │ 设备 │ 联系人 │ KYC │
└────┬──────┬──────┬──────┬──────┬──────┬──────┬──────────────┘
     │      │      │      │      │      │      │
     ▼      ▼      ▼      ▼      ▼      ▼      ▼
┌─────────────────────────────────────────────────────────────┐
│                    特征衍生层                                 │
│ CTF条件聚合 │ Word2Vec │ 图网络 │ 目标编码 │ 社交传播 │ 行为统计 │
└────┬──────┬──────┬──────┬──────┬──────┬─────────────────────┘
     │      │      │      │      │      │
     ▼      ▼      ▼      ▼      ▼      ▼
┌─────────────────────────────────────────────────────────────┐
│                    特征筛选层                                 │
│  LGB重要性筛选 → WOE稳定性评估 → VLM分布监控 → 特征入选      │
└─────────────────────────┬───────────────────────────────────┘
                          ▼
┌─────────────────────────────────────────────────────────────┐
│                    模型融合层                                 │
│  子分模型 → 与基线模型融合 → 增益评估 → 上线                  │
└─────────────────────────────────────────────────────────────┘
```

## 三、7大特征衍生技术路线

### 路线1：CTF条件聚合框架（征信数据挖掘）

**对应代码目录**：`征信数据挖掘/基础统计类特征/`、`征信数据挖掘/特征联动/`

**核心思想**：对用户的多笔征信订单，按 **条件(C) × 目标(T) × 聚合函数(F)** 的笛卡尔积系统化生成特征。

**条件(C)分组**（14列、6组）：

| 条件组 | 条件列 | 取值示例 |
|--------|-------|---------|
| 日期 | `gap_days_start_bin`, `gap_days_due_bin` | `betw360and720`, `ge720`, `betw180and360` |
| 机构 | `inst_scale_type`, `inst_ec_risk_type`, `inst_inst_risk_type`, `inst_type` | `top_inst`, `middle_inst`, `tail_inst`, `ec_high_risk_inst` |
| 逾期 | `pastdue_days_bin`, `maxdue_days_bin` | `Current`, `DPD1-89`, `DPD180+` |
| 状态 | `contract_type`, `contract_status_type` | `CreditCard`, `Paidoff`, `ActiveFacilities`, `Writtenoff` |
| 金额 | `initial_creditlimit_bin`, `debit_balance_bin` | `le3w`, `betw3wand10w`, `ge500w` |
| 期限 | `loan_term_bin` | `le30`, `betw30and90`, `betw90and180`, `ge720` |

**目标(T)**：

| 目标类型 | 具体指标 |
|---------|---------|
| 订单数量 | `num_order` |
| 在贷金额 | `sum_outstanding`, `mean_outstanding`, `max_outstanding` |
| 日期差 | `min/max/mean_start_days_diff`, `min/max/mean_due_days_diff` |
| 本金 | `sum_principal`, `mean_principal`, `max_principal` |
| 机构多样性 | `num_inst`, `max_order_inst_order_ratio` |
| 期限 | `mean/max/min_loan_term` |
| 订单间隔 | `mean/max/min_start_date_gap` |
| 逾期 | `max_dpdmax`, `mean_dpdmax`, `max_dpd`, `mean_dpd` |

**阶数机制**：

- **0阶**：无条件，全量订单聚合
  - 命名：`num_order`, `sum_outstanding`
- **1阶**：单条件过滤
  - 命名：`inst_scale_type=top_inst#num_order`
- **2阶**：两组条件交叉（AND逻辑）
  - 命名：`gap_days_start_bin=betw180and360&inst_scale_type=top_inst#sum_outstanding`

**跨窗口趋势特征**：
- 时间窗口：3M / 6M / 12M / 24M / 全量
- 窗口间差值/比值 → 捕捉信用行为趋势

**特征联动（跨源交叉）**：

将AFPI/CBI征信订单与APP安装数据关联，核心增加了APP安装状态维度：

- `install_status`：`install_before`(借前已安装)、`install_in_180d`(借后180天内安装)、`install_after_180d`、`not_installed`、`not_covered`
- `cnt_competitor_install_0d/3d/7d/30d`：借款日前N天安装竞品APP数量
- `has_new_install_L7D_before_due`：还款日前7天是否新装借贷APP

命名示例：`install_status_install_before___cnt_competitor_install_7d_wT6M_sum`

### 路线2：Word2Vec嵌入（APP + 赌博APP）

**对应代码目录**：`APP/APP_embedding/`、`赌博类app编码/`

#### 2.1 APP嵌入

**训练流程**：
1. 将每个用户的APP安装列表构建为"句子"（按安装时间排序，排除系统APP和自身APP）
2. 训练Skip-gram Word2Vec模型
   - `vector_size=64`, `window=8`, `sg=1`, `negative=15`, `min_count=10`, `epochs=10`
   - 训练语料：2024-08至2025-10样本，1/3抽样

**APP风险分级（目标编码前置）**：
1. 计算每个APP的WOE = APP逾期率 / 基准逾期率（自身APP `com.fintopia.idnEasycash.google`）
2. 按月计算逾期率，归一化后计算CV（变异系数）评估稳定性
3. 按WOE分箱 + 稳定性分层 → 12个风险等级：
   - `level_1`~`level_9`（按WOE阈值[0.6, 0.7, 0.8, 0.9, 1.2, 1.3, 1.4, 1.5]分箱）
   - `level_unstable_lowrisk`, `level_unstable_highrisk`（CV>0.3，按WOE<1或≥1细分）
   - `level_very_unstable`（CV>1）

**特征聚合**：

| 维度 | 取值 | 说明 |
|------|------|------|
| 时间窗口 | all / last180 / last30 | 全量、近180天、近30天 |
| 聚合方式 | max / mean / sum | 逐元素聚合 |
| 风险分类 | 无 / 12个level | 按APP风险等级分组 |

- 无分类：3窗口 × 3聚合 × 64维 = **576特征**
- 有分类：12级 × 3窗口 × 3聚合 × 64维 = **6,912特征**

命名模式：`{level}_{window}_{agg}_index{i}`，如 `level_5_last180_mean_index32`

#### 2.2 赌博APP嵌入

- `vector_size=100`, `window=5`, `min_count=20`
- 对赌博类APP做增量训练强化嵌入语义
- 3窗口 × 3聚合 × 100维 = **900嵌入特征**
- 额外：KMeans聚类(k=5) + 逾期用户余弦相似度

### 路线3：图网络特征（App-App图 + User-User图）

**对应代码目录**：`APP/APP图网络特征挖掘v2.1/`、`APP/App图网络特征挖掘/`、`APP/User-User_APP关系图谱/`

#### 3.1 App-App图网络（v2.1版，~69特征）

**图构建**：
1. 构建二部图：用户→APP（排除系统APP、时间穿越数据）
2. 筛选APP：用户数≥100且安装率≤10%
3. 共现矩阵 `C = M^T @ M`，计算Jaccard相似度
4. 节点属性：贝叶斯平滑逾期率 = `(bad_users + overall_rate × 50) / (total_users + 50)`
5. 边属性：`risk_lift = co_bad_rate / overall_rate`
6. Top-20近邻剪枝
7. **每月PIT滚动快照**（v2.1核心改进）

**6层特征体系**：

| 层 | 特征数 | 类型 | 说明 |
|----|--------|------|------|
| L0 覆盖度 | 5 | 安全层 | `total/matched/unknown_app_count/ratio` |
| L1 节点风险 | 7 | 风险层 | `app_risk_{mean/max/min/std/wmean}`, `high_risk_app_{count/ratio}` |
| L2 二跳传播 | 5 | 风险层 | `risk_2hop_{mean/max/std/high_ratio/lift}` |
| L3 边级风险 | 6 | 风险层 | `risk_lift_{mean/max/std/wmean}`, `high_risk_edge_{cnt/ratio}` |
| L4 社区检测 | 6 | 混合层 | Louvain社区`{entropy/hhi/dominant_ratio/risk_mean/risk_max/n_communities}` |
| L5 子图拓扑 | 4 | 安全层 | `subgraph_{node_count/edge_count/density/clustering}` |
| L6 SVD嵌入 | 36 | 安全层 | 32维SVD + `emb_norm_mean` + `emb_cos_{mean/min}` + `emb_to_global_cos` |

**生产部署**：`AppGraphFeatureOperator` 类支持实时推理，加载3个冻结文件（node_attrs, edge_attrs, global_meta）。

#### 3.2 User-User相似图（~40特征）

**图构建**：
1. IDF加权：稀有APP获得更高权重 → `IDF(app) = log(N_users / doc_freq)`
2. 用户间IDF-Jaccard相似度 → Top-20近邻图
3. 分块多进程计算（2000用户/块，64 workers）
4. OOT用户通过倒排索引高效查询训练集近邻

**4层特征**：

| 层 | 前缀 | 特征数 | 核心特征 |
|----|------|--------|---------|
| 邻居目标聚合 | `nb_` | 15 | `nb_bad_weighted_by_sim`(相似度加权逾期率), `nb_bad_top3/5_sim` |
| 图结构 | `uu_` | 8 | `pagerank`, `core_number`, `clustering_coeff`, `degree_centrality` |
| APP类别分布 | `cat_` | 12 | `finance/lending/game_ratio`, `risk_level_mean/max`, `google_entropy` |
| 时间特征 | `rush_`/`app_` | 5 | `rush_install_count_7d`(突击安装), `avg/min/max_install_age_days` |

**防泄漏**：邻居聚合使用LOO（Leave-One-Out），训练时排除自身标签。

### 路线6：目标编码/风险Lift编码

**对应代码目录**：`设备型号_省份_行业编码/`

#### 6.1 编码方法

1. 筛选样本量 > 100 的类别
2. 计算每个类别的逾期率
3. **Lift = 类别逾期率 / 全局逾期率**
   - Lift > 1.0 → 高于平均风险
   - Lift < 1.0 → 低于平均风险
4. 分别对 mob1/mob3/mob6 计算

#### 6.2 输出特征（9个）

### 路线7：AUTH交叉特征（鉴权特征挖掘）

**对应代码目录**：`特征盘点/鉴权特征挖掘/`

#### 7.1 专家规则特征

| 特征类型 | 方法 | 说明 |
|---------|------|------|
| 教育×行业匹配度 | 19×8人工定义矩阵 | 每个(行业,教育)组合一个0-1匹配度分数 |

#### 7.2 多头查询加速度特征（~770个）

基于AdvanceAI多平台查询数据，计算**短期/长期查询数量比值**：

- **7类机构**：`bank`, `cateB`, `ksp`, `loanSupermarket`, `multiFinance`, `ojk`, `otherFintech`
- **3个时间段层级**：
  - period1: `[1h, 3h, 6h, 12h, 24h]`
  - period2: `[3d, 7d, 14d, 21d, 30d]`
  - period3: `[60d, 90d, 180d, 360d]`
- **跨层级比值**：短期查询数 / 长期查询数
- Phone和ID两个维度 × 7机构 × 多期组合 ≈ **770个ratio特征**

#### 7.3 复合风险信号

- `duotou_6h`：6小时内是否有多平台查询
- `score1`：声称未申请贷款但6小时内有多头查询 → 高风险信号

## 四、特征筛选标准流程

```
全量特征
  │
  ▼
Step 1: 缺失率筛选（<98%缺失率）
  │
  ▼
Step 2: LGB重要性筛选
  ├─ 训练包含全量变量的LightGBM模型
  └─ 保留 perc > 0（或 > 0.01）的特征
  │
  ▼
Step 3: WOE稳定性评估
  ├─ 训练集WOE分箱 → OOT集评估
  ├─ 计算训练集vs OOT的Kendall相关系数
  │   ├─ corr ≤ 0     → 不稳定，删除
  │   ├─ 0 < corr ≤ 0.5 → 临界，人工审核WOE趋势图
  │   └─ corr > 0.5   → 稳定，保留
  └─ 输出稳定特征列表（vlmwoe_stable_vars_xxx.txt）
  │
  ▼
Step 4: 单特征AUC评估（AUC > 0.52 保留）
  │
  ▼
Step 5: VLM分布监控（线上特征分布漂移检测）
```

### 5.2 增益评估维度

| 评估维度 | 指标 | 说明 |
|---------|------|------|
| 区分度 | KS / AUC | OOT集上对比基线模型的提升 |
| 风险斜率 | decile bad rate slope | 分位数逾期率单调性 |
| 交叉分箱 | baseline bin × new score bin | 新特征在基线模型各分箱内的额外区分能力 |
| 时间稳定性 | 按月/周的KS/AUC | 模型性能跨时间的稳定性 |
| 子群体 | 白户 vs 非白户 | 不同人群上的差异化表现 |

### 5.3 融合方式

1. **Score融合**：`baseline_score + 子分score` → 训练融合LightGBM
2. **Score+原始特征融合**：`baseline_score + 子分score + 原始特征子集` → 进一步提升
3. 评估增量贡献（边际增益）
