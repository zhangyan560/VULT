# 图网络特征方法论

基于用户App安装列表构建图网络，提取用户风险特征的完整方法论。核心设计原则：

1. **彻底防止数据穿越**：Label Echo防护 + Leave-One-Out Buffer
2. **图结构/风险分离**：Behavior Graph（无标签）+ Risk Projection Layer（轻标签）
3. **生产级工程落地**：Top-K Neighbor Index + Graph Embedding
4. **可解释性优先**：所有特征必须能映射为业务含义

---

## PIT时间切片

### 构建原则

**仅使用训练集数据构建图，严格遵循 Point-in-Time 原则。**

```
训练集用户数据（first_done_date <= SPLIT_DATE）
    ↓
按进件日期分组
    ↓
为每个日期构建快照图
    ↓
PIT过滤：mob1_7_mature_date <= target_date - LEAVE_ONE_OUT_BUFFER
    ↓
计算边属性（仅基于共现，不使用target）
    ↓
Top-K 裁剪（每节点只保留 Top20 neighbors）
    ↓
快照图缓存
```

### 数据预处理流程

```
原始数据
    ↓
1. 训练集/测试集分割（按日期或比例）
    ↓
2. 展开App信息为用户-App对
    ↓
3. 过滤系统应用（sysApp=True）
    ↓
4. 时间防穿越（firstInstallTime <= first_done_ts）
    ↓
5. PIT标签过滤（mob1_7_mature_date <= 目标日期 - buffer）
    ↓
6. 长尾和超级App过滤
   - MIN_APP_USERS = 5（安装用户数>=5）
   - MAX_APP_INSTALL_RATIO = 0.75（安装率<=75%）
    ↓
7. 测试集App对齐（只保留训练集App）
    ↓
预处理完成
```

### 结构/标签更新分离（P1级优化）

在生产环境中，图的构建分为两层：

1. **图拓扑结构更新（每天做）**：新用户安装了哪些App，`app_stats['total_users']` 增加。用于计算Jaccard和分母，实时安全。
2. **风险标签更新（滞后做）**：只拿 `mature_date <= yesterday` 的成熟批次数据去更新 `bad_users` 计数。

这样图的结构（共现/安装人数）可以实时/T+1更新，而风险属性（Bad Rate/Risk Lift）严格遵循表现期成熟原则。

### 训练集/测试集快照选择

| 阶段 | 使用数据 | 快照获取方式 | LOO处理 |
|------|---------|-------------|----------|
| 训练集图构建 | 仅训练集数据 | 训练集所有日期 | 排除当前批次 |
| 训练集特征提取 | 训练集+训练集快照 | 对应 `first_done_date` | 排除当前用户 |
| 测试集特征提取 | 测试集+训练集快照 | 最后一个训练集快照 | 排除当前用户 |

```python
def get_snapshot_for_user(user_date, user_id, snapshot_dict, train_dates):
    if user_date in train_dates:
        snapshot = snapshot_dict[user_date]
    else:
        latest_train_date = max(train_dates)
        snapshot = snapshot_dict[latest_train_date]
    return snapshot.with_loo_exclusion(user_id)
```

## 贝叶斯平滑

### App坏账率计算（推荐）

```
bad_rate = (bad_users + overall_bad_rate * C) / (total_users + C)
```

- `C = 50`（BAYES_CONFIDENCE_THRESHOLD）
- 相当于"相信50个样本的大盘表现"
- 稀疏样本向大盘收敛，密集样本向真实值收敛

### 与拉普拉斯平滑对比

| 方案 | 稀疏样本行为 | 密集样本行为 | 优点 |
|------|-------------|-------------|------|
| 贝叶斯平滑（推荐） | 向 overall_bad_rate 收敛 | 向真实值收敛 | 自适应大盘、有统计意义 |
| 拉普拉斯平滑（不推荐） | 向 1/10 = 10% 收敛 | 向真实值收敛 | 简单，但硬编码不合理 |

### 高危App判定（带小样本过滤）

```
is_high_risk = (bad_rate > overall_bad_rate * 1.5) AND (bad_users >= 10)
```

小样本过滤说明：某App只有2个用户安装、1个逾期，坏账率50%会被误判为高危。增加绝对数量下限 `MIN_BAD_USERS_FOR_RISK = 10` 可过滤噪音。

## LOO防Label Echo

### 问题

App Risk 本质是用户风险的聚合函数。如果用一群人的逾期率去预测这群人中某个人的逾期概率，就构成了循环论证（Label Echo），导致AUC虚高、稳定性差。

### 解决方案：Leave-One-Out Buffer

计算 App risk 时排除当前批次用户，并设置时间缓冲：

```python
def get_app_risk_with_loo(app, target_date, user_batch_ids, buffer_days=30):
    pit_cutoff = pd.to_datetime(target_date) - pd.Timedelta(days=buffer_days)

    # 获取历史用户（排除当前批次）
    historical_users = [
        u for u in app_stats[app]['user_list']
        if u not in user_batch_ids and u <= pit_cutoff
    ]

    if len(historical_users) == 0:
        return 0.0

    historical_bad_rate = sum(1 for u in historical_users if get_target(u) == 1) / len(historical_users)

    # 贝叶斯平滑
    C = BAYES_CONFIDENCE_THRESHOLD
    smooth_rate = (historical_bad_rate + overall_bad_rate * C) / (1 + C)

    return smooth_rate
```

参数配置：
- `buffer_days = 30~60`（推荐值）
- `BAYES_CONFIDENCE_THRESHOLD = 50`

## 双层图设计

### 架构总览

```
+----------------------------------------------------------+
|                    用户-App 二部图                         |
|              (User-App Bipartite Graph)                   |
+----------------------------+-----------------------------+
                             |
+----------------------------v-----------------------------+
|                STEP1: Behavior Graph（无标签）             |
|  - 只使用共现关系，禁止 target 参与                        |
|  - 边类型：co-install, TFIDF, PMI                         |
+----------------------------+-----------------------------+
                             |
+----------------------------v-----------------------------+
|              STEP2: Graph Embedding（周期训练）             |
|  - Node2Vec / GraphSAGE / LightGCN                       |
|  - 输出：App Embedding Store                              |
+----------------------------+-----------------------------+
                             |
+----------------------------v-----------------------------+
|         STEP3: Risk Projection Layer（轻标签）             |
|  - 只在最后聚合时使用标签                                  |
|  - 使用 Leave-One-Out Buffer 防止 Label Echo               |
+----------------------------+-----------------------------+
                             |
+----------------------------v-----------------------------+
|              STEP4: User Feature Engineering               |
|  - Embedding聚合 + Risk加权（谨慎） + 统计特征             |
+----------------------------+-----------------------------+
                             |
+----------------------------v-----------------------------+
|                   STEP5: Risk Model                       |
+----------------------------------------------------------+
```

### 核心原则

| 原则 | 说明 | 违反后果 |
|------|------|----------|
| **标签不参与拓扑** | Risk label 只在最后投影时使用 | Label Echo, AUC虚高稳定性差 |
| **Leave-One-Out** | App risk 计算排除当前批次 | 同批次互相污染 |
| **Top-K裁剪** | 每个App只保留 Top20 neighbors | 图指数膨胀, TB级存储 |
| **Embedding优先** | 优先用低维表示替代高维统计 | 高频噪声, 稳定性差 |

### 边属性（无标签版本）

#### TF-IDF权重（推荐）

```python
tfidf_weight(A, B) = co_occurrence(A, B) / (log(total_users_A) * log(total_users_B))
```

长尾高危借条App之间的关联会被极度放大，提升欺诈捕捉率。

#### PMI（Pointwise Mutual Information）

```python
PMI(A, B) = log( (P(A,B) * P_total) / (P(A) * P(B)) )
```

#### Jaccard相似度（不推荐用于边权重，可用于行为子图阈值）

```python
Jaccard(A, B) = |A n B| / |A u B|
```

对超级大App不够友好，建议使用TF-IDF或PMI替代。

### 双图子策略（局部子图特征）

为避免"孤岛效应"（正常App稀释了风险信号的拓扑失真），局部子图特征采用双图策略：

1. **行为共现子图**：Jaccard > 0.05 即连边，用于计算 `graph_density`、`clustering_coefficient`
2. **高危诱导子图**：剔除度数为0的正常App节点，只保留高危边连接的子图，用于计算 `subgraph_node_count`、`subgraph_edge_count`、`high_risk_edge_count`

## Graph Embedding

### 为什么需要Embedding

| 特征类型 | 稳定性 | 生产经验 |
|---------|--------|----------|
| 手工统计图特征 | 中等 | 高频噪声 |
| Graph Embedding | 高（3~5倍） | 稳定且鲁棒 |

### 推荐算法

#### Node2Vec（中小图推荐）

```python
node2vec = Node2Vec(
    G,
    dimensions=128,
    walk_length=30,
    num_walks=10,
    p=1, q=1,
    workers=4,
)
model = node2vec.fit(window=10, min_count=1)
```

#### GraphSAGE（大图推荐）

```python
class GraphSAGE(torch.nn.Module):
    def __init__(self):
        super().__init__()
        self.conv1 = SAGEConv(in_channels, hidden_channels)
        self.conv2 = SAGEConv(hidden_channels, out_channels)

    def forward(self, x, edge_index):
        x = self.conv1(x, edge_index).relu()
        x = F.dropout(x, p=0.5, training=self.training)
        x = self.conv2(x, edge_index)
        return x
```

#### LightGCN（二部图推荐）

适用于User-App二部图场景，性能更优。

### 训练策略

| 策略 | 频率 | 说明 |
|------|------|------|
| 周期性训练 | 每周/双周 | 平衡稳定性与新鲜度 |
| 增量训练 | 每日 | 只训练新节点，旧节点微调 |
| 固定版本 | 按需 | 保证完全一致性（用于回溯） |

**重要**：Node2Vec是Transductive模型，不同快照分别训练会导致Embedding空间不一致。应在训练集最终全量图上训练一次，固定该Embedding字典，训练集和测试集都查同一个字典。

## Top-K裁剪

### 问题

图会指数膨胀。假设 50k apps，边理论上 1.25B pairs。

### 解决方案

每个App只保留 Top-K neighbors（K=20 推荐）：

```python
def build_top_k_index(app_pairs, top_k=20):
    top_k_index = {}
    for app_a, neighbors in app_pairs.items():
        sorted_neighbors = sorted(neighbors.items(), key=lambda x: x[1], reverse=True)
        top_k_index[app_a] = sorted_neighbors[:top_k]
    return top_k_index
```

### 效果对比

| 方案 | 存储成本 | AUC影响 |
|------|---------|---------|
| 全图 | 100% | 基准 |
| Top20 | <3% | 几乎不变 |

### 工程实现：稀疏矩阵相乘（P0级优化）

共现矩阵计算必须使用稀疏矩阵乘法，不能用Python字典双重循环（会OOM）：

```python
import scipy.sparse as sp

# 1. 构建 用户-App 稀疏矩阵 A (User x App)
A = sp.csr_matrix((values, (user_indices, app_indices)), shape=(num_users, num_apps))

# 2. App-App 共现矩阵 C = A^T * A
C = A.T.dot(A)

# 3. C.diagonal() = 每个App的安装人数
# 4. C[i,j] = App i和App j的共同安装人数
```

离线架构中，快照图的生成应完全交给 Spark/SQL 或 scipy.sparse 批处理完成。

## 部署架构

### 离线训练 + 在线推理

```
离线训练（批处理）：
  Behavior Graph Build (Spark)
    -> Top-K Index
  Graph Embedding (周期训练)
    -> Embedding Store
  Risk Projection (LOO)
    -> App Risk Store

在线推理（实时）：
  User Apps List
    -> Embedding Aggregation (查询 Embedding Store)
    -> Feature Projection
    -> User Features
```

### 数据存储结构

```sql
-- App Embedding表
CREATE TABLE dwd.app_embedding_di (
    app_package STRING,
    embedding_version BIGINT,
    embedding_array ARRAY<DOUBLE>,
    train_date DATE,
    dt STRING
) PARTITIONED BY (dt);

-- Top-K邻居索引表
CREATE TABLE dwd.top_k_neighbor_index_di (
    app_package STRING,
    neighbor_app STRING,
    neighbor_rank INT,
    edge_weight DOUBLE,
    graph_version BIGINT,
    dt STRING
) PARTITIONED BY (dt);

-- App风险表（带LOO保护）
CREATE TABLE dwd.app_risk_di (
    app_package STRING,
    risk_date DATE,
    loo_buffer_days INT,
    bad_users BIGINT,
    total_users BIGINT,
    bad_rate DOUBLE,
    is_high_risk BOOLEAN,
    dt STRING
) PARTITIONED BY (dt);
```

### 部署策略

| 阶段 | 任务 | 频率 |
|------|------|------|
| 离线训练 | Behavior Graph构建 | 每日 |
| 离线训练 | Graph Embedding训练 | 每周/双周 |
| 离线训练 | Risk Projection计算 | 每日 |
| 在线推理 | Embedding Aggregation | 实时 |
| 在线推理 | Feature Projection | 实时 |

### 版本化快照（推荐方案）

每次更新图时保存完整快照，历史特征用对应版本快照计算。

- 全局递增版本号，每次更新记录
- 建议保留最近90天的快照
- 按 `dt + version` 双分区
- 定期清理过期快照
