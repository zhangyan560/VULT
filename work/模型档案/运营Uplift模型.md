# 运营Uplift模型（WA投放）

**周会日期**：2026-04-03
**对应旧版**：2025_mbf复贷WA泳道下单uplift模型

---

## 一、背景

### 运营策略现状

WA（WhatsApp）是当前核心营销触达渠道，线上每 7 天发送成功 PV **~70 万条**，覆盖首贷未完件、首贷已完件、复贷续借、复贷在贷四类客群。复贷续借体量最大，占总发送量 **81%+**。

当前策略存在三个核心问题：

```
问题一：客群粗放
  仅按额度（>=50W / >=200W）、评级（A1-H）等固定规则筛人，
  未能识别"发 WA 后会额外产生放款"的个体差异。

问题二：策略单一
  固定 T+N 节点触达，不同用户收到相同消息策略，
  无差异化强度分级。

问题三：效果难以量化
  大部分场景无长期对照实验，
  WA 带来的增量放款难以与自然转化剥离。
```

### 业务需求

通过 **Uplift 模型** 识别"受 WA 触达影响而额外放款"的用户（Persuadables），替代人工规则圈人，提升单位 WA 配额的放款增量效率。

### 迭代目标对比

| 维度 | 当前状态 | 迭代目标 |
|------|--------|---------|
| **圈人方式** | 人工选节点，经验驱动 | 模型驱动 + 规则引擎 |
| **决策时机** | 离线日批为主 | 离线日批 + 实时事件结合 |
| **客群粒度** | 粗分4大类，无进一步细分 | 多维度精细化分层 |
| **场景丰富度** | 12个基础场景，类型单一 | 按用户实时行为自适配场景 |
| **反馈循环** | 人工维护，成本高 | 自动化评估与规则迭代 |
| **效果度量** | 难以回收实验数据 | 分层增量、校准、漂移全面监测 |
| **人工职责** | 全流程介入 | 主要负责红线、签核与异常熔断 |

## 二、建模方向与优先级

会议确认：**分场景独立建模，不区分发送节点**，优先级如下：

| 优先级 | 场景 | 建模说明 |
|--------|------|---------|
| **P0（优先）** | **复贷续借** | 体量最大，实验数据最充足；不考虑首贷历史；可关注次新用户 |
| P1 | 首贷（过件后） | 可纳入用户在未完件阶段的行为特征辅助建模 |
| P2 | 首贷未完件 | 需结合机型等维度筛选优质用户进行建模 |

> 当前聚焦：复贷续借场景，打通全流程建模闭环后再扩展其他场景。

## 三、模型实现方式

### 3.1 模型目标

Uplift 模型回答的核心问题：**「发 WA 相对不发 WA，能额外带来多少放款？」**

```
Uplift(user) = P(放款 | 发WA, X) - P(放款 | 不发WA, X)

  > 0  →  Persuadables（说服型）：WA 带来增量，重点触达
  ≈ 0  →  Sure Things / Lost Causes：发不发影响不大，节省配额
  < 0  →  Do-Not-Disturbs（反效果）：主动屏蔽
```

### 3.2 模型架构

采用 **单模型 T-Learner + LGBM** 架构：

```
训练阶段：
  在合并样本（treatment=1 发WA / treatment=0 不发WA）上，
  分别训练：
    M1：P(放款 | treatment=1, X)  → score_1
    M0：P(放款 | treatment=0, X)  → score_0

  Uplift 分 = score_1 - score_0

推理阶段：
  对当日复贷续借用户打分 → 按 Uplift 分排序 → 高分优先触达
```

> 旧版验证结论：单模型在 OOT Lift 上略优于双模型（1.60 vs 1.56），因此继续采用单模型。

### 3.3 分场景建模的变化

| 维度 | 旧版 | 新版（本次） |
|------|------|------------|
| 场景范围 | 复贷续借 + 复贷结清混合建模 | **分场景独立建模**（复贷续借优先） |
| 节点区分 | 不区分节点（T2H / 月维度等合并） | 同旧版，**不区分节点** |
| WA 归因 | 按实验 treatment 标记 | **末次归因规则**（见第五节） |
| Y 定义 | T+3放款率（主）/ T+7累计 | **Y1**（金额加权分类）+ **Y2**（金额回归） |

## 五、样本定义

### 5.1 样本单元

**样本粒度：人天（person-day）**

业务推理场景是"每天早上跑批，决定这个人今天要不要发 WA"，决策单元天然是**人 × 天**，因此训练样本粒度也应对齐为人天，保证训练与推理的一致性。

**人天去重规则**：同一用户同一天多条 WA 记录，按"转化 > 未转化"合并为一条。

### 5.2 正负样本与对照组定义

| 类别 | treatment | 条件 |
|------|-----------|------|
| **正样本** | 1（发WA） | WA 投放后 **1天 / 3天** 内有放款，且表现期内至少一笔放款 |
| **负样本** | 1（发WA） | WA 投放后 **1天 / 3天** 内无放款，表现期内均无放款 |
| **对照-正** | 0（不发WA） | 同期未收到 WA 的用户，表现期内有放款 |
| **对照-负** | 0（不发WA） | 同期未收到 WA 的用户，表现期内无放款 |

**对照组来源（按质量优先级）**：

| 来源 | 可用性 | 说明 |
|------|:------:|------|
| 全局 Holdout（随机分配的永久不触达用户） | 优先 | 不发WA是随机决定的，可比性最强 |
| 各场景实验空白组 / 泳道对照组 | 次选 | 随机分配，质量次之 |
| 分配到测试组但当天未发WA的用户 | 不可用 | 未发原因非随机（频控、业务规则等），有选择偏差 |

**同一用户可以跨期出现在不同组**：人天粒度下，同一用户在不同时间点可分别贡献 treatment 样本（收到WA的天）和 control 样本（未收到WA的天），需满足两个条件：
1. **静默窗口隔离**：用户收到 WA 后开启 N 天静默窗口，窗口内该用户不计入任何其他样本
2. **触达历史作为特征**：在特征中加入历史WA接收次数、距上次WA间隔天数等

> 旧版样本规模参考：~254万条（发WA 177万 vs 不发WA 74万），训练集部分欠采样。

### 5.3 WA 归因规则（末次归因原则）

用户在短期内可能被多次投放 WA，需确定哪一次 WA 应归因为驱动该次放款的触发点。

```
放款事件发生时，往前追溯：

优先级1：有 WA 已读记录
  └── 取放款前最近一次「已读」WA 作为归因样本

优先级2：无已读，有发送成功记录
  └── 取最近一次「发送成功」WA 作为归因样本

优先级3：无已读 & 无发送成功
  └── 取最近一次「请求发送」WA 作为归因样本

其余 WA 记录：
  └── 不被归因，正常作为训练样本保留，标记 Y = 0
```

**多条 WA 的处理示例**：

```
Day1: user1 收到 WA_1  → treatment=1
Day2: user1 收到 WA_2  → treatment=1
Day3: 放款

末次归因结果：
  Day2 的 WA_2 → Y = 1（归因命中，最近一条）
  Day1 的 WA_1 → Y = 0（未被归因，强制置0，但保留为训练样本）
```

> [!important]
> 未归因的 WA 保留为 Y=0 而不是直接丢掉。若直接丢弃，模型只见到"发WA→转化"的样本，会高估所有 WA 的效果。保留 Y=0 能让模型学到：不是每条 WA 都能驱动转化，只有在关键时机触达才有效。

**归因局限性**：末次归因假设"最近那条 WA 是因果触发点"，但更早的 WA 可能才是真正种草的。工程上的缓解手段：将"距上次WA间隔天数"作为特征输入，长期可考虑 Shapley value 归因。

## 六、目标变量（Y）

### Y1：分类目标（主）

- **定义**：WA 投放后近 1天 / 3天 内是否至少有一笔放款（0 / 1）
- **样本加权**：以**用户总放款金额加权**——金额越大权重越高
- **意义**：优先学习高价值用户的转化规律，兼顾转化率与放款体量
- **对应旧版**：旧版采用 T+3放款率，不加权；新版引入金额权重

### Y2：回归目标（辅）

- **定义**：表现期内放款金额求和
- **意义**：直接优化放款金额，识别高额用户的增量贡献
- **说明**：Y2 可单独训练或与 Y1 联合多任务训练后融合打分

## 七、特征工程

### 7.1 特征模块总览

| 特征类别 | 代表特征 | 备注 |
|---------|---------|------|
| **用户基础** | 信用评级(A1-H)、KTP状态、注册时长、历史借款次数 | 稳定特征，T-1离线 |
| **贷款状态** | 可借额度、在贷/结清、历史逾期次数、最近还款时间、额度变化幅度 | T-1离线 |
| **行为活跃** | 7/14/30天登录频次、App打开时段偏好、还款及下单间隔天数 | 滑动窗口统计 |
| **APP埋点** | 社交APP活跃信号、金融APP行为信号、App安装/卸载delta特征 | 需埋点挖掘 |
| **触达历史** | 历史WA接收次数、点击率、历史实验组归属 | 防偏差积累 |
| **产品偏好** | 曾申请的产品类型、利率敏感度（日利率>0.2%分层）、券使用记录 | 与当前策略对齐 |
| **实时信号（阶段2）** | 当日登录事件、还款事件、风控跑批结果 | 在线特征，秒级更新 |

### 7.2 APP 埋点特征挖掘

旧版模型中，**APP 类埋点特征**在特征重要性中排名较高（如 Facebook/Orca 最后更新距今天数）。

```
活跃设备信号
  └── 社交 APP（Facebook / Instagram / WhatsApp / TikTok）
      ├── 最后一次更新 / 打开距今天数
      ├── 安装 / 卸载行为（新安装 gap 最小值）
      └── 各类目 APP 安装数变化（delta 特征）

金融 APP 行为信号
  └── 金融类 APP 新安装、使用频率
      ├── 可反映用户对借贷产品的活跃意愿
      └── 结合 FINANCE 类目 delta 特征
```

> 旧版验证结论：实时 APP 特征（当日信号）+离线特征子分的双模块组合，OOT AUC 从 0.748 提升至 0.758，说明埋点特征有增量价值。

### 8.1 技术指标

|------|------|----------|

### 8.2 业务指标（在线实验）

- **核心**：增量放款笔数 / 增量放款金额（实验组 vs 对照组 A/B 实验）
- **辅助**：WA 退订率、投诉率（负向指标，防止过度打扰）

## 十二、多处理组 Uplift 模型设计

### 问题的本质差异

本场景是一个**多处理组 + 多维度**问题：

```
Treatment 空间 = 场景(4) x 节点(N) x 消息类型(3~4) x 强度(弱/中/强)
≈ 数十到上百个 treatment 组合
```

**不能直接套用传统 uplift 的原因**：无法为每个 treatment 组合单独建模（样本量不足、维护成本爆炸）；不同实验的对照组定义不统一；treatment 之间有优先级与互斥关系。

### 两阶段建模架构

```
┌───────────────────────────────────────────────────────────────────┐
│  第一阶段：全局触达 Uplift 模型（触达 vs 完全不触达）               │
│                                                                   │
│  输入：用户特征 X + 场景特征 S + 节点特征 T                        │
│  输出：uplift分 = P(放款|发WA) - P(放款|不发WA)                   │
│  决策：uplift > 阈值 → 进入触达名单                                │
└───────────────────────────────────────────────────────────────────┘
                              │ 确认触达
                              ▼
┌───────────────────────────────────────────────────────────────────┐
│  第二阶段：消息类型最优动作选择（Multi-arm 策略模型）               │
│                                                                   │
│  输入：用户特征 X + 场景 S                                         │
│  候选动作：{utility, marketing, chatbot, push}                    │
│  输出：各动作的期望增量，选最优                                     │
│  方法：Multi-treatment uplift 或 Contextual Bandit                │
└───────────────────────────────────────────────────────────────────┘
```

**核心设计思路**：将 treatment 空间解耦为特征维度

- 维度1（触达 vs 不触达）→ uplift 模型核心，统一全局样本池
- 维度2/3（触达时机/场景）→ 编码为特征输入
- 维度4（消息类型）→ 确认触达后的条件策略选择

### 统一样本池构建

**全局 Treatment 与 Control 定义**：

- Treatment：凡是被发送了任意一条 WA 的用户（覆盖所有12个场景）
- Control 优先：全局 Holdout（5~10%永久不触达）；次选：各场景实验空白组

**跨场景样本不可比问题**：通过倾向得分加权（IPW）校正，训练倾向模型 P(被触达|X,S,T)，用倾向得分对样本加权后合并建模。

**特征编码**：

```
场景特征 S：贷款生命周期阶段、是否KTP、客群价值分层
节点特征 T：触发事件类型、距离事件发生的时间、距离上次触达的时间间隔、当日是否已被其他场景触达
```

### 第二阶段：多消息类型最优动作选择

| 方案 | 适用场景 | 说明 |
|------|---------|------|
| **Multi-treatment T-Learner** | 样本充足时 | 每种消息类型一个响应模型，选 Uplift 最大的 |
| **Contextual Bandit** | 样本不足或探索阶段 | LinUCB / Thompson Sampling 平衡探索与利用 |
| **Treatment Embedding** | 长期方向 | treatment 做 embedding 与用户特征联合输入，支持新 treatment 泛化 |

### 落地路径

| 阶段 | 模型形态 | 核心工作 | 解决的问题 |
|------|---------|---------|-----------|
| **MVP** | 单场景二元 Uplift | 选取复贷续借跑批T2H场景，建经典 T-Learner | 跑通建模流程，验证增量效果 |
| **扩展** | 跨场景统一 Uplift | 用 IPW 合并多场景样本，场景/节点特征化 | 样本不足、跨场景泛化 |
| **优化** | 两阶段模型 | 触达决策模型 + 消息类型 Bandit | 同时优化"发不发"和"发什么" |
| **成熟** | Treatment Embedding | 统一 embedding 模型 | 新消息类型冷启动 |

## 十三、Agent 自动化运营体系设想

```
                    ┌─────────────────────────────────────┐
                    │          自动化运营 Agent 系统        │
                    └─────────────────────────────────────┘
                                      │
          ┌───────────────────────────┼───────────────────────────┐
          ▼                           ▼                           ▼
┌──────────────────┐       ┌──────────────────┐       ┌──────────────────┐
│   感知层 Agent    │       │   决策层 Agent    │       │   执行层 Agent    │
│ (数据 & 特征)     │       │ (Uplift 模型)     │       │ (WA 发送 & 频控) │
└──────────────────┘       └──────────────────┘       └──────────────────┘
          │                           │                           │
          ▼                           ▼                           ▼
┌──────────────────┐       ┌──────────────────┐       ┌──────────────────┐
│   监控层 Agent    │◄─────►│   学习层 Agent    │◄─────►│   治理层 Agent    │
│ (效果评估)        │       │ (模型迭代)        │       │ (规则 & 熔断)     │
└──────────────────┘       └──────────────────┘       └──────────────────┘
```

### 分阶段落地计划

| 阶段 | 目标 | 关键交付 | 依赖条件 |
|------|------|---------|---------|
| **Phase 0（现在）** | 打通数据与实验 | 配置长期对照实验；统一日志格式 | 各场景补配对照组 |
| **Phase 1（3个月）** | 离线 Uplift 模型上线 | T-Learner 模型跑批；日批名单替代人工选节点 | 足量实验样本（>=3个月） |
| **Phase 2（6个月）** | 决策层 Agent 接入 | 打分服务化；频控统一入口；消息类型自动选择 | 在线特征工程基础设施 |
| **Phase 3（9个月）** | 实时事件触发 | 感知层+执行层 Agent；秒级响应还款/登录事件 | 实时消息队列（Kafka/Flink） |
| **Phase 4（12个月）** | 全闭环自动化 | 监控+学习+治理 Agent 联动；模型自迭代 | 评估体系稳定；签核流程制度化 |

### 应用价值

| 价值维度 | 具体收益 |
|---------|---------|
| **放款增量** | 精准识别 Persuadables，提升单位 WA 配额产出的放款金额 |
| **成本节省** | 减少无效 WA 发送量（WA 按条计费），降低渠道成本；减少用户退订 |
| **运营效率** | 从人工每周选节点→模型每日自动圈人，释放运营人力 |
| **迭代速度** | 闭环自动评估+重训，从月级迭代压缩到周级甚至天级 |
| **风险管控** | 自动检测模型漂移、异常熔断 |
| **可解释性** | Uplift 分档可映射到「发不发、发几条、何时发」 |
