# Claude Code - 上下文窗口管理

> [!important] 核心原则
> **主动管理**比被动等待自动压缩效果好 10 倍。在质量下降之前干预，而不是等到崩溃。

## 三大核心命令

### `/context` — 查看当前状态

```bash
/context
```

显示当前上下文窗口占用百分比和估算费用。**养成随时查看的习惯。**

### `/clear` — 彻底清除

```bash
/clear
```

- **效果：** token 计数归零，完全擦除对话历史
- **使用时机：** 完成一个独立任务，准备转入完全不相关的新任务
- **代价：** 丢失所有上下文（可用 `/resume` 恢复历史会话）
- **适合：** 任务边界清晰的场景

### `/compact` — 压缩摘要

```bash
/compact
# 或带指令压缩（保留关键内容）：
/compact "保留关于登录逻辑和 JWT 实现的讨论"
```

- **效果：** Claude 生成对话摘要，以摘要作为新起点，释放大量历史 token
- **使用时机：** 上下文占用达到 **60–70%** 时手动执行
- **代价：** 多次压缩后（通常 3–4 次）可能丢失部分细节
- **适合：** 任务还未结束，但历史太长的场景

## /clear vs /compact 决策树

```
当前任务完成了吗？
    ├── 是 → 准备做新任务？
    │         ├── 是（完全无关） → /clear
    │         └── 否（还在同一项目） → /compact
    └── 否 → 上下文用了多少？
              ├── < 60% → 继续工作
              ├── 60–75% → /compact（带指令）
              └── > 75% → 立即 /compact（质量已开始下降）
```

> [!warning] 不要等到 95% 才处理
> 系统默认在 95% 自动压缩，但此时输出质量已明显下降。建议在 60–70% 主动干预。

## 会话生命周期管理

### 标准会话节奏

```
开始新任务
    ↓
/clear（清除上一个任务的历史）
    ↓
说明任务背景 + 关键约束（一条详尽的消息）
    ↓
工作...
    ↓
达到 60–70% → /compact "保留 [关键内容]"
    ↓
继续工作...
    ↓
任务完成 → /clear
```

### 跨会话续接技术（GitHub Issue 法）

适用于上下文必然会满的**大型任务**：

1. 在 Claude 中生成详细计划（包含已完成进度）
2. 让 Claude 将计划写入 GitHub Issue：
   ```
   用 gh CLI 创建一个 issue，包含我们的实施计划和已完成的进度
   ```
3. 执行 `/clear` 完全清空上下文
4. 在新会话中：
   ```
   读取 GitHub issue #42，继续完成未完成的部分
   ```

这样可以在保持精确上下文的同时，将 token 成本重置为 0。

## 模型切换策略

根据任务复杂度选择合适模型，避免用 Opus 做简单任务：

```bash
/model opus    # 架构设计、复杂推理、困难 bug
/model sonnet  # 日常编码（最常用）
/model haiku   # 简单子代理任务、格式化、研究摘要
```

**努力度（Effort）调节：** 用方向键左右切换：
- 低努力度：简单任务（省 token）
- 高努力度：仅在处理极复杂逻辑时使用

## 实时监控：状态栏（Status Line）

启用状态栏后，可以实时看到：
- 当前 token 占用百分比
- 当前使用的模型
- 估算费用

在 `settings.json` 中启用或使用 `/statusline` 命令配置。

## Plan Mode（计划模式）

在执行复杂任务前，先进入计划模式：

```
/plan
```

Plan Mode 让 Claude 先规划方案，**不执行任何代码修改**。确认方案正确后再执行，避免因错误实现路径浪费大量代码生成 token。
