# Claude Code - 提示词压缩技巧

> [!tip] 核心思想
> 每个字都有成本。写给 Claude 的每条指令，都会在每次消息中被重读。

## CLAUDE.md 的最佳写法

`CLAUDE.md` 是每条消息都会加载的"系统宪法"，写法至关重要。

### 黄金法则：索引而非仓库

**错误做法：** 将整个 API 文档、大型配置文件、详细教程塞进去
**正确做法：** 只保留核心规范，用"路标"指向其他文件

```markdown
# 错误示例（肿胀型）
## API 参考
[粘贴了500行API文档]
## 数据库Schema
[粘贴了完整Schema定义]

# 正确示例（索引型）
## 规范
- 技术栈：React + TypeScript + Supabase
- 代码风格：ESLint + Prettier（见 .eslintrc）
- API 文档：需要时读取 docs/api-reference.md
- DB Schema：需要时读取 prisma/schema.prisma
```

### CLAUDE.md 结构模板

```markdown
# 项目：[名称]
## 技术栈（1-3行）
## 构建命令（2-5行）
## 编码规范（5-10条）
## 95%置信度规则（防止跑偏）
## 外部参考文档路径（索引）
## 应用学习（让Claude自动记录错误）
```

**目标：保持在 200 行以内**

### 95% 置信度规则

在 `CLAUDE.md` 中加入：

```
在行动前，如果置信度低于95%，必须先提问，不要假设。
```

这条规则能防止模型因理解偏差而生成错误代码，减少因反复修改产生的额外 token。

## 提示词写法优化

### 批量化（Batching）

**错误习惯（浪费2/3 token）：**
```
消息1：帮我检查 auth.js 有没有 bug
消息2：顺便看看性能问题
消息3：再看一下安全漏洞
```

**正确做法（一条搞定）：**
```
请检查 auth.js 文件，关注三个方面：
1. 潜在 bug（重点看 verifyUser 函数）
2. 性能问题
3. 安全漏洞
```

三条消息 vs 一条消息：节省约 **66%** 的历史重读开销。

### 外科手术式引用

**错误（让 Claude 自行探索）：**
```
帮我找一下代码里的 bug
```

**正确（精确指向）：**
```
检查 src/auth/verifyUser.ts 第47-89行，
具体看 validateToken() 函数的边界条件处理
```

使用 `@文件名` 语法直接指向具体文件，避免 Claude 盲目遍历整个代码库。

### 关键词陷阱：避免触发高消耗思考

| 危险词            | 触发效果    | 替代词                  |
| -------------- | ------- | -------------------- |
| `think`        | 高强度思考模式 | `consider`           |
| `think harder` | 更高消耗思考  | `evaluate carefully` |
| `ultra think`  | 极高消耗    | `review thoroughly`  |

> [!warning] 慎用"think"
> 在提示词中随意出现"think"会触发高 token 消耗的推理模式。日常任务用中性词替代。

## 说明动机的价值

不要只告诉 Claude **做什么**，要告诉它**为什么**：

```
# 低质量提示
把这个函数改成异步的

# 高质量提示
把这个函数改成异步的。
原因：我们迁移到了新的数据库驱动，所有 DB 调用现在必须是 async/await 模式，
这样才能配合新的连接池管理（见 db/pool.ts）。
```

第二种写法虽然多了几个 token，但 Claude 理解动机后能主动发现代码库中其他需要同步修改的地方，减少反复 follow-up，**总体节省更多 token**。

## 使用视觉输入节省描述 Token

Claude Opus 4 的视觉处理能力很强。用截图传达信息比用文字描述 UI 状态更高效：

- 截图报错信息 > 手动描述错误堆栈
- 截图 UI 设计图 > 用文字描述布局需求
- 截图代码高亮 > 大段粘贴代码

## Prompt Stashing（提示词暂存）

使用 `Ctrl+S` 可以暂存当前正在写的提示词，处理插入任务后自动恢复，避免重新输入浪费时间和 token。
