# Claude Code - 工作流设计与避坑

> [!note] 最高层次的优化
> 单条提示词的优化是战术，工作流设计才是战略。系统化的工作流能将总体成本降低 60–80%。

## 常见误区（反省清单）

### 误区1：碎片化提问（最常见）

**错误认知：** 小问题多发几条消息，这样每条都很短
**实际后果：** 每条消息都要重读全部历史，三条消息 = 三倍历史重读成本

**修正：** 将相关问题批量成一条详尽消息

---

### 误区2：超时后继续对话

**错误行为：** 离开电脑 30 分钟回来后，直接继续上一条话题发消息
**实际后果：** Prompt Cache 5 分钟后失效，全额重新处理所有历史

**修正：** 长时间离开前执行 `/compact` 或 `/clear`

---

### 误区3：随意使用"think"关键词

**错误做法：** `"请仔细思考（think harder）这个架构设计..."`
**实际后果：** 触发高强度推理模式，消耗远超必要的 token

**修正：** 日常使用 `consider`、`evaluate`、`review` 替代

---

### 误区4：重复上传相同文件

**错误做法：** 在同一会话中多次用 @ 引用同一文件
**实际后果：** 每次引用都把文件内容再次加入上下文

**修正：** 同一会话中文件已在上下文里，直接引用函数名即可

---

### 误区5：等待自动压缩（95%）

**错误认知：** 系统会自动压缩，不用管它
**实际后果：** 95% 时输出质量已明显下降（"Loss in the Middle" 效应）

**修正：** 在 60–70% 时手动 `/compact`，并给出保留指令

---

### 误区6：所有任务都用 Opus

**错误认知：** 最贵的模型 = 最好的结果
**实际后果：** 日常编码用 Opus 的成本是 Sonnet 的 5–10 倍，但效果差异很小

**修正：** 按任务复杂度选模型（见 [[03-上下文窗口管理#模型切换策略]]）

## 高效工作流设计（三层架构）

### Tier 1：基础卫生习惯（立即可用）

- [ ] 每次新任务前 `/clear`
- [ ] 相关问题合并为一条消息发送
- [ ] 会话开始时 `/mcp` 断开不需要的服务器
- [ ] 启用状态栏实时监控 token 占用
- [ ] 使用 Plan Mode 规划复杂任务

### Tier 2：上下文工程（进阶）

- [ ] 保持 `CLAUDE.md` ≤ 200 行，索引模式
- [ ] 在 60–70% 时主动 `/compact`（带保留指令）
- [ ] 精确引用文件和函数（外科手术式）
- [ ] 缓存意识：长时间离开前压缩/清除

### Tier 3：架构级优化（高阶）

- [ ] 子代理隔离高消耗任务（用 Haiku 跑）
- [ ] GitHub Issue 法跨会话续接大型任务
- [ ] Context Mode 虚拟化大型文件索引
- [ ] 外部工具预规划，带最终方案入场

## 大型任务的完整工作流

```
1. 外部规划阶段（Grok / ChatGPT）
   └── 初步方案讨论，不消耗 Claude token

2. 进入 Claude Plan Mode
   └── /plan + 明确需求 + 详细背景
   └── 确认多阶段实施方案

3. 写入 GitHub Issue（状态存储）
   └── gh issue create --title "计划：XX功能"
   └── 包含已完成进度 + 下一步计划

4. 执行阶段
   └── 开始实施（上下文 < 60% 正常工作）
   └── 达到 60–70% → /compact "保留 [关键内容]"
   └── 阶段完成 → 更新 GitHub Issue

5. 上下文满时续接
   └── /clear（完全重置）
   └── 新会话："读取 issue #XX，继续完成 Phase 2"
```

## CLAUDE.md 自进化配置

让 Claude 自动记录项目中的陷阱和绕行方案，避免未来会话重复踩坑：

```markdown
## 应用学习（Applied Learning）
<!-- Claude 在发现反复出现的错误或特殊绕行方案时，自动在此处追加记录 -->
<!-- 格式：[日期] 问题描述 → 解决方案 -->
```

这个板块会随使用积累形成**项目专属知识库**，每次会话都能读到，避免重复出错。
