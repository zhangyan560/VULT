# Claude Code - 工具调用与MCP优化

> [!warning] 隐形杀手
> MCP 服务器是最容易被忽视的 token 消耗源。每个服务器每条消息都会加载工具定义，完全透明，却持续消耗。

## MCP 服务器的 Token 成本

每个连接的 MCP 服务器，会在**每条消息**中加载其完整工具定义：

| 场景 | token 消耗 |
|------|-----------|
| 单个大型 MCP（如 Google Workspace） | ~18,000 token/消息 |
| 同时连接 5 个 MCP | ~90,000 token/消息（纯工具定义） |
| 同时连接 10 个 MCP | ~180,000 token/消息 |

**结论：** 连接不必要的 MCP 服务器，等于每条消息白送给服务商一大笔钱。

## 精简 MCP 的三种策略

### 策略1：按任务断开不需要的 MCP

```bash
/mcp  # 查看当前连接的 MCP 服务器
```

会话开始时，主动断开当前任务不需要的服务器。例如：
- 写代码时不需要 Google Calendar MCP
- 做数据分析时不需要 GitHub MCP

### 策略2：CLI 替代 MCP

对于很多场景，直接让 Claude 使用 CLI 工具比加载 MCP 服务器更轻量：

| 任务 | MCP 方案（重） | CLI 方案（轻） |
|------|--------------|--------------|
| GitHub 操作 | GitHub MCP | `gh` CLI |
| 文件搜索 | Filesystem MCP | `find`/`grep` |
| 数据库查询 | Database MCP | 直接 SQL CLI |

### 策略3：Context Mode 虚拟化层

对于需要处理大型文件（日志、CSV、代码库索引）的场景，使用 **Context Mode** 类 MCP 服务器：

**工作原理：**
```
大型文件（50,000+ 行）
    ↓ 索引到本地 SQLite 数据库
    ↓
Claude 通过搜索查询获取片段
    ↓
每次只读取相关的几百行
```

**效果：** 最高减少 **99%** 的相关 token 消耗（Better Stack 视频案例）

## 子代理（Sub-agents）的成本管理

### 成本现实

子代理启动时需要加载自己完整的上下文，成本是标准会话的 **7–10 倍**。

**但子代理有独特价值：** 隔离高消耗任务，保护主会话的上下文清洁。

### 子代理使用权衡

| 场景 | 是否值得用子代理 | 原因 |
|------|----------------|------|
| 阅读大型外部文档/API | ✅ 值得 | 防止垃圾信息污染主上下文 |
| 研究任务（结果摘要即可） | ✅ 值得 | 主代理只需最终结论 |
| 简单格式化任务 | ❌ 不值 | 用 Haiku 直接处理更省 |
| 核心业务逻辑编写 | ❌ 不值 | 需要完整上下文理解 |

### 子代理成本优化

```
主代理（Sonnet）
    ↓ 分配任务
    ├── 子代理1（Haiku）: 查阅 API 文档 → 返回摘要
    ├── 子代理2（Haiku）: 搜索相关代码模式 → 返回结果
    └── 汇总结果 → 主代理基于摘要实现功能
```

**关键：** 为子代理指定更便宜的模型（Haiku），主代理保留 Sonnet/Opus。

## 并行工具调用

指示 Claude 同时发起多个工具调用（并行读取文件、搜索等），而不是串行执行：

```
在 CLAUDE.md 中添加：
"使用并行工具调用同时处理多个独立任务（同时读取多个文件、搜索等）"
```

并行调用不减少 token 总量，但减少**消息轮数**，从而降低历史重读的累积成本。

## 文件读取优化

### 避免读取超大文件

- Claude 默认文件读取上限：**25k token**
- 终端输出默认上限：**30,000 字符**（可在 settings.json 中调高到 150,000）

### 精确引用而非全量读取

```
# 避免
读一下整个 utils.ts 文件

# 推荐
读取 utils.ts 中的 formatDate 和 parseDate 函数（约在第45-89行）
```
